# §0 序言

## §0.1 目的

在 BSV 上构建一个全新的、生产级的、无荷官的多人**扑克**平台，它：

1. 作为单一自包含应用运行，可在 **Windows 桌面**和 **web** 上启动，在本地/regtest 模式下游玩无需任何外部服务；
2. 让玩家能够**互相发现并建立连接**并就座于牌桌；
3. 实现**真正的扑克** — 从一个可完整游玩的变体开始，并通过一个干净的游戏模块接口扩展至标准变体集合；
4. **无荷官且非托管**：没有服务器持有牌堆，没有运营方决定结果，每个游戏事件都是一笔已签名的 BSV 交易，牌桌状态是有效交易集合的确定性函数；
5. **组合**已有的 BSV 原语（`prof-faustus` 栈 — §2），而非重新发明它们；
6. 按 §13 中的标准来工程化，并按 §14 中的义务来测试。

本文档是设计（此处）与构建（Claude Code）之间的契约。

## §0.2 不可妥协的原则

这些原则统辖后续所有章节。违反其中任何一条都是构建缺陷，而非风格偏好。

- **P1 — 仅 BSV。** 仅限 Genesis 升级后的 BSV 共识。没有 BTC 代码、没有 BTC 假设、没有 Lightning 式的脚本内惩罚构造。见 §6.2。
- **P2 — 确定性（相对于一个已商定的交易排序）。** 给定有效交易集合的一个*已商定*排序以及规则集哈希，`table_state = f(ordered_valid_txs, ruleset)`，且持有同一已排序集合的两个诚实客户端会推导出逐字节相同的状态。引擎在*给定*一个排序时是确定性的；它绝不依赖于挂钟到达时间或某节点先看到哪个数据包。**它声称什么、不声称什么（修正 RT-01 M1）：** 在区块确认之前，全局"有效交易集合"并非普遍达成一致 — 各内存池互不相同，交易可能被逐出或被重组（§8.4）。因此确认前的*安全性*并非仅靠确定性提供，而是靠收敛机制：双路径传播（§8.3）、确定性冲突规则（§8.5）以及超时默认分支（§6.4）。关于排序的分歧由 §8.5 解决，而非由引擎解决。重组/逐出由恢复/超时路径（§6.4）处理；离开牌桌的价值仅在区块确认时才最终确定（§5.7）。引擎的确定性正是使重放（§12.3）和跨客户端一致性（§14.2）可核查的根本；它并不声称所有客户端始终持有相同的未确认集合。
- **P3 — 无可信庄家、无可信运营方。** 洗牌是分布式的；中继仅作传输和索引，绝不充当真相之源（§8.1）。
- **P4 — 每个可操作状态都有两个出口。** 一个协作分支（玩家行动）和一个超时默认分支（玩家不行动）。时间是一个显式的协议参与者。任何牌桌都不会被某个缺席或恶意的参与者冻结（§7、§6.4）。
- **P5 — 弃牌不亮牌。** 弃牌放弃主张并将该玩家移出争夺，而不披露其隐藏的牌。只有仍在场的争夺者才亮牌，且只亮摊牌所要求的部分（§5.6、§4.6）。
- **P6 — 零捏造。** 每个常量都可追溯到一个来源（被引用的文档、实测值，或声明的 `TRACKED ASSUMPTION`）。没有任何数字是为了让另一个数字自洽而编造的。一个造成内部不一致的硬编码数值是重大缺陷。
- **P7 — 显式假设。** 每条假设都出现在规范中，并且凡是约束代码之处也出现在源代码中。隐藏的假设被视为最严重的一类失败，因为它们假装不存在，却污染了下游的一切。
- **P8 — 不过度声称。** 一条陈述所断言的内容不得超过该构造在结构上所能保证的范围。只有在一个未花费-到期失效输出机制确实强制执行的地方，才允许说"撤销是一个链上事实"；"无信任（trustless）"不会被无条件使用 — 信任面被逐一列举（§18）。
- **P9 — 测试在解释器内部失败。** Script 花费通过开启 Genesis 规则的真实 BSV Script 解释器进行验证。否定测试在解释器内部失败，而非在手写的守卫中失败。签名抽查不能作为运行脚本的替代品（§14.3）。
- **P10 — 可复现的证据。** 每个被报告的数字都能由一条命令从已提交的测试向量中重新生成，该命令在任何不匹配时以非零退出（§14.5）。

## §0.3 范围

**在范围内（本项目）：**

- 平台核心（引擎、密码学、交易层、钱包、网络、持久化）。
- 作为游戏模块的扑克变体：德州扑克（Texas Hold'em）、奥马哈（Omaha）、七张梭哈（Seven-Card Stud）、五张抽换（Five-Card Draw）、Razz。（21 点单独追踪 — §0.5 D7 — 因为无荷官的 21 点与扑克并非同一个对称的心智扑克问题，绝不能在同一模型下偷渡进来。）
- 一个自包含运行时（"the VM"，§10），它捆绑一个 regtest 模式下的本地 BSV 节点、中继和客户端，使整套系统在无任何外部依赖的情况下启动。
- 基于一个共享核心之上的 Windows 桌面外壳与 web 外壳（§11）。
- 玩家发现与牌桌匹配撮合（§8）。
- 通过 `bonded-subsat-channel` 进行微下注集成，用于游戏内的亚聪价值（§2.2），在早期阶段为可选且置于一个开关之后。

**不在平台项目范围内，作为后续轨道提供（非第 1 阶段）：**

- 可撤销加密内容 NFT（基于 `overlay-broadcast` 构建，§2.4）— 一条独立的产品轨道；平台暴露集成接缝（§15.7），但在核心扑克阶段不交付任何 NFT 产品。
- TEE 封装的密钥托管 — 按方向指示为**可选**。架构将托管接口（§9.3）设计为可插拔，以便日后可在不改变协议的前提下添加 TEE 后端。下文没有任何阶段要求 TEE。
- 真钱运营、KYC/AML、受监管的博彩。系统默认为 regtest；主网（mainnet）需在一个显式的研究开关之后主动启用（§10.5）。游戏内代币默认不携带任何外部货币价值。这不是一个受监管的博彩产品，本规范也不假装它是（P8）。
- 移动端（iOS/Android）原生外壳。web 外壳对移动端做了响应式适配；原生移动端是后续轨道。

## §0.5 待决决策（这些均可被覆盖；为不阻塞构建而取了默认值）

这些是真实的抉择。已为默认值附上理据，以便设计可以推进（遵循"去设计、勿停滞"的指示）。
每一项都做了标记，以便可在一处被覆盖。

- **D1 — 首个可游玩游戏。** 默认：regtest 上的**单挑（2 人）无限注德州扑克**。理据：德州扑克是心智扑克的典范目标（两张私有底牌 + 共享公共牌锻炼隐藏、选择性亮牌以及一棵真实的下注树）；单挑将多方边池的复杂性从首个里程碑中移除，同时仍锻炼整条流水线（熵承诺/揭示、分布式洗牌、加密发牌、下注 FSM、摊牌、结算、超时）。覆盖时设置 `phase1.game`。
- **D2 — 平台的牌桌规模。** 默认：2–9 座；第 1 阶段固定为 2；第 3 阶段启用 6-max；第 4 阶段启用 9-max。覆盖时设置 `table.maxSeats` 的取值范围。
- **D3 — 下注结构。** 默认：先无限注（No-Limit）；底池限注（Pot-Limit）与固定限注（Fixed-Limit）作为同一接口之后的下注结构策略（§5.4）。可按游戏模块覆盖。
- **D4 — 第 1 阶段的发现机制。** 默认：一个小型托管**中继**（Go），用于在线状态 + 牌桌发现 + 直接消息扇出，外加一个 **LAN 自动发现**模式，用于零服务器的同网络游玩。带 NAT 穿透（§8.7）的完整互联网 P2P 在第 5 阶段。理据：一个仅作传输的中继（绝不充当真相之源，P3）是在不损害确定性的前提下跨互联网"找到其他玩家"的最短路径。覆盖时设置 `discovery.mode`。
- **D5 — "VM" 的解释。** 默认："FULL VM" = 一个可复现的自包含运行时镜像（容器镜像 + 可选的打包 VM 镜像），捆绑本地 BSV 节点（regtest）、中继和客户端，端到端启动且无任何外部服务（§10）。如果在容器之外还要求一个字面意义上的虚拟机镜像（如 OVA/qcow2），那属于 `DECISION REQUIRED`，并在 §16 中增加一个打包目标；应用架构不变。
- **D6 — 复用 prof-faustus 的自包含节点。** 默认：复用 `bonded-subsat-channel` 中随附的嵌入式 BSV 节点（它已提供一个原生 P2P-wire 节点、一条按"最长工作量链"选择的工作量证明区块头链、一个基于数据库的区块/UTXO 存储、一个在原始替换规则下检测冲突的内存池，以及一个 HD 钱包）作为平台的本地链后端（§10.2），而非新建一个节点。覆盖时设置 `chain.backend`。
- **D7 — 21 点（Blackjack）。** 默认：推迟到独立轨道。理据：标准 21 点是玩家对庄家；无荷官的 21 点需要一套不同于对称扑克洗牌的结算与隐藏模型，必须按其自身条件设计，而不能强塞进扑克流水线（P7 禁止偷渡其差异）。覆盖时将其移入一个带有自身设计章节的阶段。
- **D8 — 早期阶段的货币/筹码语义。** 默认：**不具任何外部货币价值**的游戏币筹码，在 regtest 上结算，因此第 1–4 阶段不存在任何博彩监管面。覆盖是一个具有法律后果的严肃决策，超出本文档的权限范围。

## §0.6 术语表（权威；全文精确使用这些术语）

- **心智扑克（Mental poker）** — 在网络上进行一种纸牌游戏，没有可信庄家，使得洗牌可验证，且任何低于全体的联盟都无法在揭示之前得知一张隐藏的牌。
- **分布式/可验证洗牌** — 每个玩家依次对牌堆加密并置换，使最终顺序是所有秘密置换的复合，对任何单个玩家而言都是未知的；每一步都可被证明是一次真正的置换。
- **承诺-揭示（Commit-reveal）** — 玩家先发布 `H(value)`（具约束力但不披露），随后再揭示 `value`；防止在看到他人的"随机性"后再选择自己的。
- **组合公钥（Combined public key）** — 对一张牌/元素，将每方各一个公钥做 EC 点加；对应的组合私钥是各方标量之和。它是否在某一处被重建取决于签名模式（§4.3：模式 A 在揭示时重建；模式 B 不重建）。花费锁定到组合密钥的 UTXO 是平仓/结算原语，而非发牌（GB2616862；§4.3）。
- **洗牌密钥（Shuffle key）** — secp256k1 上的一个 EC 点 `P' = (s, ±√(s³+7))`，其 x 坐标即私钥 `s`，公钥为 `P = s·G`；它给出一个双向的点↔标量关系，用于在集合内部混合私钥（GB2616862；§4.2）。
- **加密牌 UTXO（Encrypted-card UTXO）** — 一张牌在链上表示为一个 UTXO，其公开身份为 `(deck_id, card_serial, ciphertext_commitment, locking_script)`，其牌面值在揭示之前一直被隐藏（§4.5、§6.3）。
- **协作分支 / 超时默认分支** — 每个可操作状态都有的两个出口（P4）。
- **公平对局交易（Fair-play transaction）** — 一笔交易，其锁定脚本在脚本内证明：某方所承诺的私钥确实能推导出他们所使用的公钥和加密洗牌密钥；发送了不匹配值的作弊者将丧失绑定到这笔交易上的资金（GB2616862 §6.5.2；§4.7）。
- **规则集哈希（Ruleset hash）** — 规范序列化后游戏配置（变体、筹注额、盲注/底注、超时、座位数、牌堆格式）的 `H`。被绑入每一笔交易，使某个分支无法在不同的上下文下被重放（§6.3、§5.2）。
- **中继（Relay）** — 传输 + 索引服务。加速收敛与重广播；绝不充当真相之源（P3、§8.1）。
- **The VM** — 捆绑节点 + 中继 + 客户端的自包含运行时镜像（§10），依 D5。
- **Genesis 升级后（Post-Genesis）** — Genesis 升级之后的 BSV 共识状态：没有区块大小上限、恢复了脚本能力，以及一些不同于 BTC 的特定操作码语义 — 尤其是 `OP_CHECKLOCKTIMEVERIFY`/`OP_CHECKSEQUENCEVERIFY` 的空操作（no-op）状态（§6.2）。

# §1 产品定义

## §1.1 用户体验到什么

一名玩家可以：在 Windows 上安装或打开 web 应用；创建或导入一个钱包；设置或加入一张牌桌
（选择变体、筹注额、座位数、超时配置）；看到牌桌（座位、轮到谁、公共牌/明牌、底池、
本地解密的自己的隐藏手牌、计时器以及不作为的确切后果）；采取合法动作（看牌、下注、跟注、
加注、弃牌；在有该机制的变体中抽换/停牌）；进入摊牌或在无人争夺时获胜；并看到确定性结算。
玩家采取的每个动作都是一笔已签名的交易；UI 会显示何时需要签名以及它承诺了什么。

## §1.2 面向用户的能力（REQ 登记册种子 — 完整 ID 见 §13）

- REQ-PROD-001 钱包创建/导入；密钥托管可插拔（§9）。
- REQ-PROD-002 大厅：列出/创建/加入牌桌；显示其他玩家的在线状态（§8）。
- REQ-PROD-003 规则集 + 筹注配置，并显示一份带哈希的配置（§5.2）。
- REQ-PROD-004 牌桌视图：座位、轮次、公共牌/明牌、底池、余额、计时器、超时默认文案（§11.4）。
- REQ-PROD-005 本地私有手牌检视（解密自己的牌；绝不在托管边界之外向 UI 进程暴露密钥）（§9.3、§11.5）。
- REQ-PROD-006 准确说明正在签署什么的签名提示（§11.6）。
- REQ-PROD-007 弃牌不亮牌动作（P5）。
- REQ-PROD-008 摊牌时只揭示所需的部分（§5.6）。
- REQ-PROD-009 确定性结算显示 + 最终余额（§5.7、§6）。
- REQ-PROD-010 记录誊本导出 + 确定性离线重放（§12.3）。
- REQ-PROD-011 重连/恢复进入一张进行中的牌桌（§8.6）。
- REQ-PROD-012 清晰的研究/regtest 横幅；主网置于显式开关之后（§10.5）。

## §1.3 首个发行版的非目标

没有多桌锦标赛、没有抽水/运营方经济、没有 NFT 市场、没有真钱运营、没有原生移动端、
不要求 TEE。首个发行版的存在是为了在 regtest 上为一个变体证明完整流水线端到端可用，
并具备真实的玩家发现。

# §2 依赖栈与集成契约

平台消费已有的 BSV 原语。本节定义平台向每个仓库所要求的**契约** — 接口、输入/输出、
不变式，以及平台所依赖的失败模式。Claude Code 拥有这些仓库；集成工作将平台的适配器（§15）
接线到每个仓库的实际 API。凡某仓库的具体 API 与下文某个契约符号不同之处，由适配器进行映射；
平台核心只依赖契约，绝不依赖某个仓库的内部实现（这使核心可针对伪实现进行测试，并防止某个
仓库的变更波及整个引擎）。

## §2.1 `cardtable` — 心智扑克引擎与交易原生的游戏基底

**角色：** 无荷官的游戏基底。可验证的分布式洗牌、承诺-揭示熵、加密牌 UTXO，以及协作/超时默认
交易状态机。首个目标游戏为 In-Between；平台使用其*原语*，而非其游戏。

**契约 `CT`（被消费）：**

- `CT.entropy.commit(secret) -> commitment` / `CT.entropy.reveal(commitment, secret) -> bool` — 用于洗牌随机性的承诺-揭示（§4.1）。
- `CT.shuffle.*` — 在一个 N 方集合上进行的可验证分布式洗牌，为每张牌产生组合公钥以及各方的加密洗牌密钥（§4.2–§4.4）。
- `CT.card.*` — 加密牌表示 `(deck_id, card_serial, ciphertext_commitment, locking_script)`；隐藏、本地解密、准备揭示证明、弃牌时交出（§4.5、§4.6）。
- `CT.fsm.*` — 已承诺的游戏状态转换，其中每个状态 UTXO 都有一个协作后继和一个超时默认后继；记录誊本以及离线再验证（§7）。
- `CT.tx.*` — 交易原生的事件发射与确定性回退图（§6）。

**平台所依赖的不变式（必须成立，否则平台的保证不成立）：**

- INV-CT-1 没有任何单方得知洗牌顺序（秘密置换的复合）。
- INV-CT-2 在揭示之前，一张隐藏的牌只能由其持有者解密。
- INV-CT-3 每个游戏状态 UTXO 恰好有两个有效后继（协作、超时）。
- INV-CT-4 时间锁在交易层面强制执行，绝不通过脚本内的 CLTV/CSV（§6.2）。*（这是一条贯穿全栈的不变式；见 §6.2。）*
- INV-CT-5 在其自身输出中零捏造 / 声明假设。

**视为既定的状态（依作者所述）：** 协议已完整规范；非链上密码学与仿真层可运行；早期构建阶段
已实质就位；完整的多牌游戏部分完成；首个游戏为 In-Between。因此平台**不**假设 `cardtable`
中已存在德州扑克；平台在 `CT` 原语之上构建各扑克变体（§5、§7）。

## §2.2 `bonded-subsat-channel` — 亚聪微支付 + 自包含节点

**角色（两种不同用途）：**

1. **微下注** — 以选定粒度 `k` 进行游戏内筹注的亚聪价值移动，通过最大余数对账 `Q*` 进行整聪的链上结算，并设有一个固定的一聪反作弊保证金，在广播过期状态时被罚没。用于后续阶段链下的细粒度筹码移动（§5.7、§9.4）。在第 1–2 阶段为可选且置于开关之后。
2. **自包含本地节点** — 其嵌入式 BSV 系统（原生 P2P-wire 节点、按最长工作量的 PoW 区块头链、基于数据库的区块/UTXO 存储、原始替换规则下检测冲突的内存池、HD 钱包）作为平台 VM 的本地链后端（§10.2），依 D6。

**契约 `BS`（被消费）：**

- `BS.node.*` — 启动/停止一个本地 regtest 节点；提交 tx；查询 UTXO/输出点状态；观察冲突/双花尝试；区块/区块头查询（§8.4、§10.2）。
- `BS.channel.open/transfer/close/contested` — 带 `k` 粒度与 1 聪保证金的微支付通道生命周期（§5.7）。
- `BS.reconcile.Qstar(microBalances) -> integerSatoshiOutputs` — 确定性的最大余数结算；在每个诚实方上结果相同（P2）。

**不变式：**

- INV-BS-1 链上绝不写入任何分数输出；亚聪精度只存在于链下账目中。
- INV-BS-2 所冒风险的资本是固定的（每参与者一聪），与支付金额/路径长度无关。
- INV-BS-3 时间在交易层面强制执行（原始替换下的 nLockTime/nSequence），而非脚本内（§6.2）。

## §2.3 `verifiable-accounting` 系列 — 审计证据与三式记账

**角色：** 在不泄露其他记录的前提下，将已披露的游戏/财务记录转化为可检验、有锚定的审计证据；
可选的企业级持久化/审计（§12.4）。

**契约 `VA`（被消费）：**

- `VA.merkle.prove(records, index) -> bundle` / `VA.merkle.verify(bundle) -> bool` — 针对 BSV 区块头 `merkleroot` 的 Merkle 包含证明，将信任终结于 PoW 区块头链（Layer A）。
- `VA.disclose.*` — 选择性披露：只返回被查询的记录 + 其路径；兄弟节点保持为不透明哈希（Layer B）。隐私即选择性披露本身；不额外附加任何密码学。
- `VA.chain.*` — PKI 根 + GL 密钥层级、ECDH/花费链接的已签名交易链、逐字段披露（三式记账）。

**不变式与所声明的边界（必须如实反映，不得夸大，依 P8）：**

- INV-VA-1 只确立：包含性、完整性、选择性披露、对已披露记录的算术正确性。
- INV-VA-2 **不**确立源头真实性、某事件确曾发生、分类正确性、总体完整性或法律可执行性。在其余部分自洽的账簿中于源头录入的一条虚假记录**不会**被检测到。平台凡呈现审计输出之处都必须显式呈现这一边界。
- INV-VA-3 必须保留在花费链接图中的承诺，以**活动脚本中的 pushdata 携带，而非 `OP_RETURN`**（§6.5）。

## §2.4 `overlay-broadcast` — 群组访问控制、撤销、门限托管

**角色（后续轨道；核心中暴露接缝）：** 带对数级重新密钥的密钥图（逻辑密钥层级，Logical Key
Hierarchy）广播加密、链上会话生命周期、**以未花费-到期失效输出表达的撤销**，以及门限密钥托管
（FROST/GG20）。它是可撤销加密内容 NFT（一个独立产品，不在核心扑克阶段交付）的基础，也是
平台日后若拆分底池/赢取密钥时进行**底池/赢取密钥门限托管**的归宿（§9.3）。

**契约 `OB`（被消费）：**

- `OB.keygraph.*` — 构建/重新密钥一个密钥图；以 `O(log n)` 增删成员；以经认证的密钥封装进行 wrap/unwrap（绝不用裸 XOR）。
- `OB.session.*` — 为 `k` 个会话注资；续期会花费成员输出；**过期后仍未花费 = 已撤销**（平台不得声称超出此机制所强制范围的撤销，P8）。
- `OB.custody.threshold.*` — 将一个密钥拆分为若干份额；只有达到门限才能签名/重建；任何份额（或低于门限的集合）都不泄露任何信息。
- `OB.ecies.*` / `OB.aead.*` — 对 secp256k1 公钥的 ECIES；AES-256-GCM AEAD。

**不变式：**

- INV-OB-1 在种子隔离的信令中没有任何密钥材料过线；只有位置过线。
- INV-OB-2 撤销是一个链上事实（未花费的到期失效输出），不由任何运营方决定。
- INV-OB-3 按 NPR 7150.2 + Power of Ten 分级并配有需求可追溯性矩阵 — 平台采用同一标准（§13）。

## §2.5 专利 GB2616862 — 洗牌/选择/结算原语（规范性引用）

**角色：** 洗牌与结算所构建于其上的密码学与链上机制。已通读；已记录。它在纯 BSV Script 中提供：
洗牌密钥 `P'=(s,±√(s³+7))`；通过点加得到的逐元素组合密钥；选择 = 花费组合密钥 UTXO；
两轮加密（先用共享标量，再用逐元素不同的标量）并以重排序 = 洗牌，在揭示时以逆标量还原；
OP_RETURN 阶段承诺；以及完整演算过的结算：2-of-2 注资多签、nLocktime 退款、按 locktime 分级的
下注交易、公平对局脚本，以及一个可选的 2-of-3 可信第三方强制执行。通过推广适用于 N 方和一副
52 张的牌堆（专利演算过的脚本是 2 方/3 张；将其放大是平台的工作 — §6、§4）。

**引用纪律（P6）：** 引用为 **GB2616862A**（已公布的申请，2023-09-23），除非/直到 B 阶段授权

nChain Licensing AG。任何部署许可义务都是超出本文档权限的商业事务，仅作标记，不予裁定。

## §2.6 依赖适配器原则

REQ-DEP-001 平台核心只依赖上述契约 `CT/BS/VA/OB`，通过一个适配器层（§15.8）访问。每个契约都有
一个**伪实现/内存实现**，仅用于单元测试和属性测试中的编排接线。REQ-DEP-002 任何仓库具体 API 的
变更都在其适配器中被吸收；没有任何变更传播到引擎、FSM 或 UI。REQ-DEP-003 **伪实现通过一致性套件
与现实绑定（修正 RT-01 M4）。** 每个契约一套单一的契约一致性测试套件，*同时*针对伪实现和真实适配器
运行；两者都必须通过它，因此伪实现可被证明与真实契约相符，针对伪实现的一次通过运行无法认证一个
错误的引擎。REQ-DEP-004 **安全关键**行为 — 分布式洗牌正确性、揭示单次使用、公平对局罚没、组合密钥
签名 — 都针对**真实**的 CT/密码学实现测试（链上部分通过真实的 Script 解释器，P9）；伪实现**绝不**
允许替代被测的安全属性，只可替代周边编排。

## §3.2 组件与语言

- **手牌评估热路径：** 采用一份有文档记载、有测试向量支撑的算法的 TypeScript（§5.3）；WASM 实现是同一接口之后的可选后续优化，仅当性能剖析证明其必要时才采用（不做过早优化 — Power-of-Ten 纪律）。

## §3.3 确定性核心边界

REQ-ARCH-001 引擎是输入的纯函数：`(orderedValidTxSet, ruleset) -> tableState`。
REQ-ARCH-002 引擎不执行任何 I/O、不联网、不读取时间，也不使用任何随机性，除非经由被注入、被记录
的来源（本客户端自身承诺-揭示的熵在一个确定的点抽取并记录于记录誊本中）。REQ-ARCH-003 "现在
（Now）"只作为一个显式参数进入引擎，用于超时资格评估，并按 §6.4 从链/中继锚定的高度/时间戳推导，
对于影响共识的决策绝不取自本地挂钟。正是这一边界使 P2 与 §14 的重放测试成为可能。

## §3.4 单个玩家动作的数据流（规范性时序）

1. UI 请求合法动作：`engine.getLegalActions(state, seat)`（§15.2）。
2. 玩家选择一个动作；SDK 构建对应的交易模板（§6、§15.5）。
3. 钱包/托管恰好对所承诺的字节进行签名；签名提示说明签署了什么（§9、§11.6）。
4. 动作**同时沿两条路径**传播（§8.3）：(a) 作为一笔真实交易发往本地节点/网络；(b) 经由中继/P2P 直接发给牌桌对等方，以实现快速收敛。
5. 每个对等方通过同一个引擎验证并应用该动作；相冲突的尝试由 §8.5 的确定性规则解决。
6. 状态推进；如果行动者未在截止前行动，超时默认分支变为有效，任何对等方都可推进它（§6.4）。

# §4 密码学协议：BSV 上的心智扑克

本节确定密码学方案。它将 GB2616862（§2.5）适配到 BSV 所用的 secp256k1，并使用 `cardtable`
原语（§2.1）进行编排。所有 EC 点都在 secp256k1 上；`G` 是生成元；`n` 是群的阶。点序列化对于
哈希/传输采用 SEC-1 压缩的 33 字节形式；由哈希派生的密钥材料使用 32 字节的 x 坐标（RFC 5869
惯例）。无穷远点被排除。

## §4.1 熵：承诺-揭示（无可信庄家）

每个玩家 `p` 从一个被记录的来源采样一个高熵秘密 `r_p`，发布 `c_p = H(r_p)`（承诺），随后披露
`r_p`（揭示）。组合种子 `σ = H(r_1 ‖ … ‖ r_N)`（规范方序，§4.4）为确定性洗牌参数提供种子。
REQ-CRYPTO-001 承诺后扣留揭示会触发一个确定性惩罚/回退（§5.7、§6.4）：超时罚没，或一条已承诺的
回退种子规则，绝不是某个运营方的裁定。REQ-CRYPTO-002 任何玩家都不得在观察到任何其他 `r_q` 之后
采样或更改 `r_p`；约束由承诺哈希提供，并由排序强制执行（在任何揭示开放之前，承诺即关闭）。

## §4.2 洗牌密钥与逐牌密钥（GB2616862）

对每个牌索引 `j` 和每个玩家 `p`，洗牌密钥是 EC 点
`P'_{p,j} = (s_{p,j}, +√(s_{p,j}³ + 7) mod p_field)`，其私钥 `s_{p,j}` = x 坐标，公钥
`P_{p,j} = s_{p,j}·G`。平台固定使用**正**平方根分支（协议必须承诺于一个分支；只有当分支选择泄露时，
混用分支才会将搜索空间减半 — §4.8）。逐牌私钥在托管边界内部从一个绑定到 `(gid, j)` 的逐局种子
确定性地派生（HKDF），因此一台设备只存储一个长期密钥，而非 D 个逐牌密钥（§9.2）。

## §4.3 组合公钥、牌的生命周期，以及签名模式决策

对牌 `j`，组合公钥 `Q_j = Σ_p P_{p,j}`（点加）；组合私钥 `w_j = Σ_p s_{p,j} mod n`。`Q_j` 锁定
一个 BSV UTXO。逐牌 UTXO 是该牌的所有权记录和状态机锚点。

**牌的生命周期（发牌到位置；这是扑克，不是选牌 — 修正 RT-01 B2）。** GB2616862 演算过的游戏是
最高牌取胜，玩家通过花费某牌的组合密钥 UTXO 来*选择*一张牌。扑克并非如此发牌。平台采用契约链
生命周期 `minted → drawn(position) → revealed | folded → discarded`：

- **mint（铸造）** — 洗牌后存在 52 个组合密钥牌 UTXO（§4.4），顺序对任何一方都未知。
- **draw(position)（抽取到位置）** — *发牌*操作将一个隐藏牌 UTXO 作为一次已承诺的状态转换绑定到某个座位/公共牌位置（§6.1 Deal 类）。这就是扑克发牌；它**不是**玩家的"选择"。抽取由协议针对该变体的发牌日程（§7）驱动，而非由玩家选择花费哪张牌。
- **reveal | fold（揭示 | 弃牌）** — 在摊牌时一张已抽取的牌被揭示（§4.6），或在弃牌时不揭示地交出（§4.6）。
- **discard（弃置）** — 在手牌结束时被花费进一个死手/已关闭状态。

因此对 `Q_j` 的"花费即选择（selection-by-spend）"是在揭示/摊牌与底池结算时使用的**平仓 / 结算**
原语（§5.7、§6.6） — **而非**发牌。全文都按此区分使用这些术语；"选择（selection）"绝不用来指"发牌"。

**签名模式决策（修正 RT-01 B1 — 取代此前那条无条件的"绝不在某一处重建"的声称，该声称与 GB2616862
的基础相矛盾）。** 花费一个 `Q_j` 锁定的 UTXO 需要一个在 `Q_j` 下的签名，即对 `w_j` 的知识。
两种模式，其后果白纸黑字写明：

- **模式 A — 专利字面、揭示时重建（默认，第 1 阶段）。** 依 GB2616862（图 15；第 49、59–61 页）：在揭示时，相关各方披露其逐牌标量 `s_{p,j}`；进行平仓的一方求和得到 `w_j` 并签名。**必须遵守的后果：** (i) 已披露的标量一经揭示即成为可复用的秘密材料，因此逐牌密钥是单局的且**绝不复用**（§9.2）；(ii) 锁定到组合密钥的资金**仅在该手牌窗口内**（数小时）有效，从而限定暴露（GB2616862 第 39–40 页）；(iii) 其安全论证是该专利的有界窗口论证 — **而非**"任何完整密钥从不存在"。本模式在揭示时于某一处重建 `w_j`，规范坦白地这样说明。
- **模式 B — 不重建、门限/多方签名（升级，第 2 阶段起）。** 通过一种无荷官的门限/多方 ECDSA 产生 `Q_j` 签名，使 `w_j` 从不以完整形式存在。这是 v27 的改进；它**不**特定于 BSVM，因此与 P1（仅 BSV）兼容。可经由托管后端插拔（§9.3）。只有模式 B 才可声称"无完整密钥重建"属性；模式 A 不可。

REQ-CRYPTO-008 构建在声称模式 B 的属性时绝不得呈现模式 A。当前活动模式记录在规则集中（§5.2），
并在凡向玩家展示密钥处理保证之处显式呈现（P8）。决策（D9）：第 1 阶段默认模式 A；覆盖时设置
`crypto.signingMode`。

## §4.4 分布式洗牌（两轮加密 + 置换）

依 GB2616862 §6.2 与 `CT.shuffle`：每一方按规范顺序，(i) 通过 EC 标量乘法加密每个元素，(ii) 置换
其顺序；第一轮每方使用一个共享标量，第二轮使用**逐元素不同的标量**，从而支持日后逐元素（逐牌）
的选择性揭示而不揭示整个集合。揭示时以逆标量进行还原。REQ-CRYPTO-003 **规范方序**是各方长期公钥
在 33 字节 SEC-1 压缩形式下的字典序；它被确定性地计算、作为设置状态的一部分发布，且与网络到达
顺序无关（这固定了 HKDF/排序输入，使 Dolev-Yao 重排序者无法更改派生密钥）。REQ-CRYPTO-004 每个
洗牌阶段都被承诺：`c_p = H(state_p ‖ scalars_p ‖ permutation_p)`，记录于链上（这些死端阶段承诺可
接受 OP_RETURN，§6.5），从而支持事后争议重放（§12.3）。

## §4.5 加密牌表示

一张隐藏的牌在公开层面为 `(deck_id, card_serial, ciphertext_commitment, locking_script)`；其牌面
值为 `face_j`（例如 6 位：2 位花色 + 4 位点数），承诺为 `cmt_j = H(face_j ‖ blind_j)`，并/或作为
一段 AEAD 密文携带，该密文以只有门限集合才能派生的材料作为密钥（§4.6）。REQ-CRYPTO-005 协议中途
对某张牌的密文/承诺进行替换，会被跨状态转换的字节相等性检查（§6.3）以及揭示时的 AEAD 完整性拒绝。

## §4.6 揭示（选择性的、绑定接收方的）与弃牌不亮牌

- **私有揭示（玩家自己的底牌）：** 持有者派生出足够的密钥材料以在本地解密；其他玩家只看到一个牌对象发生了移动（P5）。平台的揭示将披露绑定到特定的牌和接收方，使被披露的材料单次使用，且不可跨牌/位置/接收方重放。`[EXPAND: Part 2 — 平台所采用的确切单次使用揭示令牌构造；候选方案是绑定到 (gid, j, position, height, recipient-ephemeral-key) 的、经共识时间戳的单次使用 ECDH 令牌；此处记录为领先候选方案，尚未确定 — 在阅读 cardtable 的 reveal API 后 DECISION REQUIRED。]`
- **公开揭示（公共牌）：** 一张公共牌被*每一*方的加密所隐藏，因此*产生*该揭示是一次 **N-of-N 协作转换** — 每一方都必须释放其解密材料 — 这恰如一个玩家动作，因此它有一个**超时默认分支**（修正 RT-01 M2）：如果某方扣留，该轮街就无法被揭示，状态经由恢复路径（§6.4）解决，并在已配置之处进行保证金罚没（§2.2）。一经产生，已发布的揭示材料可被任何观察者对照承诺进行验证。单个扣留者是一次*活性*失败（手牌卡住然后走默认），绝非保密性失败。
- **弃牌：** 一笔弃牌交易证明该玩家控制其隐藏手牌输出，并将它们交出到一个死手状态而**不**披露牌面值；保留承诺；将该玩家移出争夺（P5；§6.3 弃牌脚本）。

## §4.7 公平对局强制执行（无可信第三方的反作弊）

依 GB2616862 §6.5.2：每一方在一笔公平对局交易中承诺一笔资金，只有通过在脚本内证明其所用私钥能
推导出他们发送的公钥和加密洗牌密钥，才能赎回这笔资金。发送了不匹配值的一方无法赎回并丧失这些
资金 — 从而在没有裁判的情况下使诚实对局成为理性结果。REQ-CRYPTO-006 平台在洗牌/加密之后、揭示
之前生成公平对局交易；公平对局锁定脚本由脚本内的洗牌点派生和 EC 点乘法例程构建（GB2616862 第
55–60 页）。REQ-CRYPTO-009 **公平对局的可扩放性是一个实测风险，而非一个假定的参数更改（修正
RT-01 M3）。** GB2616862 演算过的公平对局脚本是一个针对 **3 个元素和 2 方**的长串嵌套
`OP_IF`/`OP_ELSE` 结构；对于一副 52 张的牌堆和 N 方，单脚本证明可能非常庞大。Genesis 升级后的 BSV
没有脚本大小上限，故并非不可能，但其字节大小、手续费和可构建性都未经验证。在依赖单脚本方案之前，
构建必须为目标牌堆/方数**测量**公平对局脚本大小（记录于 §19.C），并且如果单脚本被证明不切实际，
必须实现回退方案 — **逐牌或逐批的公平对局交易**。在 §19.C 给出实测字节排布之前，规范不断言公平对局
"可扩放"。一个可选的 2-of-3 TTP 强制执行（GB2616862 §6.5.1）作为一个独立的、需主动启用的脚本配置
提供（§6.7）；默认不使用（P3 偏好无第三方）。

## §4.8 安全参数与所声明的限制（P8）

- REQ-CRYPTO-007 为洗牌密钥固定一个平方根分支；保持分支选择一致，并将任何会暴露两个分支之选择的协议视为使密钥搜索空间减小为原来的 1/2（GB2616862 第 39–40 页）。锁定到洗牌派生密钥的资金仅在洗牌交互窗口（数小时）内有效，从而限定暴露。
- 需要继续携带（而非掩盖）的所声明限制：N-of-N 揭示意味着任何单个扣留者都会使该次揭示停止（一次活性失败，而非保密性失败） — 由超时默认分支和保证金罚没来限定；一个门限 (t, N) 变体以抗合谋性换取活性，是一个显式的后续选项；该协议在揭示时非托管且无单一庄家，但**并非**无条件无信任 — 其信任面（secp256k1 上的 DDH；正确的链下执行；原语健全性；节点/共识完整性；托管后端）在 §18 中逐一列举。

## §4.9 每手牌的参与者集合（修正 RT-01 m1）

REQ-CRYPTO-010 洗牌是在**当前就座集合**上的 N-of-N，且**每手牌都是一次全新的 N 方洗牌**：
为每手牌生成一个新的组合种子（§4.1）、新的洗牌密钥（§4.2）、新的逐牌组合密钥（§4.3），以及一个
新的规范方序（§4.4）。REQ-CRYPTO-011 离座（sit-out）和加入**只在两手牌之间**生效；参与者集合在
手牌开始时被冻结，并作为规范方序的一个输入（使该顺序对所有人都明确定义且相同）。REQ-CRYPTO-012
进行中的牌堆**不存在部分重洗**，手牌进行期间参与方集合不变；手牌进行期间断线的玩家由超时默认/恢复
路径（§6.4）处理，而非通过重新派生牌堆来处理。

# §5 扑克领域模型

这是游戏逻辑，独立于链与密码学。它本质上篇幅很大；此处严谨规范，以便 FSM（§7）与交易（§6）
绑定到一个精确的模型。

## §5.1 牌与牌堆

标准 52 张法式牌堆。点数从高到低：A K Q J T 9 8 7 6 5 4 3 2（A 为高）；在高手牌的顺子中，
A 也可作低用于"轮（wheel）"（A-2-3-4-5），并且在 ace-to-five 低牌中为最低（§5.3.3）。花色
{c,d,h,s} 在点数上等同（扑克手牌比较中没有花色优先级）。规范牌编码：`card = rank*4 + suit`，
`rank∈0..12`（2=0 … A=12），`suit∈0..3`（c=0,d=1,h=2,s=3）。`card_serial∈0..51`。REQ-POKER-001
牌堆编码是固定的，且与绑入洗牌（§4）和交易模式（§6）的编码相同。

## §5.2 规则集及其哈希

一个 `Ruleset` 确定：变体；下注结构（NL/PL/FL）；座位数；盲注/底注日程；bring-in（梭哈）；
最小/最大买入；下注/加注的额度规则；加注次数上限（FL）；超时配置（决策超时、恢复超时）；
牌堆格式；以及货币语义（D8）。REQ-POKER-002 `rulesetHash = H(canonicalSerialize(Ruleset))`
在建桌时计算一次，向所有玩家显示，并绑入每一笔交易（§6.3），使任何分支都无法在不同的规则集下被
重放。规范序列化在 §19.A `[EXPAND: Part 3]` 中定义。

## §5.3 手牌评估（权威）

一副 5 张牌的扑克手牌先按类别比较，再按破平点数比较。评估器返回一个全序可比较值，使得
`compare(handA, handB) ∈ {-1,0,+1}` 既精确又满足传递性。REQ-POKER-003 评估器是纯的、确定性的，
并由 §19.D `[EXPAND: Part 3]` 中的测试向量目录支撑（包括所有类别边界、轮顺子以及破平/踢脚边界
情形）。

### §5.3.1 高手牌类别（从高到低）

1. 同花顺（五张连续点数、同花色；皇家同花顺是 A 高的情形；轮 A-2-3-4-5 是最低的同花顺）。
2. 四条（四张相同 + 1 张踢脚）。
3. 葫芦（三条 + 一对；先比三条，再比对子）。
4. 同花（五张同一花色；点数从高到低比较）。
5. 顺子（五张连续点数、不同花色；轮为最低）。
6. 三条（三条 + 2 张踢脚）。
7. 两对（高对、低对、1 张踢脚）。
8. 一对（一对 + 3 张踢脚）。
9. 高牌（5 张点数从高到低）。

破平时按字典序比较有序的相关点数；花色绝不破平。REQ-POKER-004 "七取五最佳"选择（德州扑克/梭哈）
枚举 C(7,5)=21 个五张子集并按 `compare` 取最大；奥马哈使用 §5.3.2 中的受约束选择。

### §5.3.2 奥马哈约束

恰好用四张底牌中的**两**张加上五张公共牌中的恰好**三**张组成手牌。REQ-POKER-005 评估器枚举
C(4,2)·C(5,3)=6·10=60 种组合并取最大；这一约束是强制的，如果误用了通用的七取五，它是 bug 的
常见来源 — 它是一条独立的、经过测试的代码路径。

### §5.3.3 低手牌评估（Razz；高低分注游戏稍后）

Razz 使用 **ace-to-five 低牌**：A 为低，顺子和同花不计为对手牌不利，最佳手牌是五张各不相同的
最低点数；可能的最佳是轮 A-2-3-4-5（"自行车，the bicycle"）。REQ-POKER-006 低牌评估返回一个
越低越好的可比较值；对子按 ace-to-five 规则受罚；其实现是一条不同于高牌评估的、有测试向量的
路径。（八优高低分注，如果某个后续变体需要它，则一手低牌只有在具备五张各不相同且 ≤8 的点数时才
有效成低 — `[EXPAND: Part 2]`。）

### §5.3.4 性能说明（Power-of-Ten 纪律）

REQ-POKER-007 评估器必须先正确，再快速。参考实现是直接的枚举，循环次数固定且有界（21 或 60），
满足有界循环规则。查找表/完美哈希优化只允许置于同一接口之后，且仅当在目标平台上的性能剖析显示
其确有必要时；它必须在整个 §19.D 向量集合上逐比特地复现参考评估器。

## §5.4 下注结构与下注状态机

REQ-POKER-008 下注是一个置于单一接口 `BettingStructure` 之后的策略：
`legalBets(state, seat) -> {check?, call?: amount, bet?: {min,max}, raise?: {min,max},
fold}` 与 `applyBet(state, seat, action) -> state'`。策略：无限注（max = 筹码量）、
底池限注（max = 当前底池 + 跟注额）、固定限注（固定的小注/大注额度，加注次数受限）。
REQ-POKER-009 下注机跟踪：每个座位的筹码量、本轮已投入、本手牌总已投入、当前需跟注额、
上一次完整加注的大小（用于最小加注合法性）、谁全下、自上一次进攻性动作以来谁已行动，以及
本轮结束条件。REQ-POKER-010 当行动回到上一位进攻者，且所有未弃牌、未全下的玩家都已跟齐当前
注（或一路看牌通过）时，下注轮结束。不足一次完整加注的全下**不会**对已行动的玩家重新开放下注，
除非它构成一次完整加注（标准规则；已测试）。

## §5.5 底池与边池

REQ-POKER-011 当一名或多名玩家以不同金额全下时，底池引擎从各座位的总投入计算出一个**主池**
和有序的**边池**。算法：将各不相同的全下/投入阈值按升序排序；对每个阈值层，每个至少投入到该层
的玩家向该池贡献该层的增量，且该池的有资格玩家集合恰好是仍在该层争夺的那些玩家；高于最高全下
的余额组成一个仅由跟齐它的玩家争夺的池。REQ-POKER-012 摊牌时每个池独立地判给有资格的最佳手牌
（平局按奇数筹码规则 §5.5.1 分配）。该算法被规范到一份参考实现的程度，并在 §19.B `[EXPAND:
Part 3]` 中附有完整演算的多人全下示例。

### §5.5.1 奇数筹码与分池规则

REQ-POKER-013 在分池时，筹码在筹码粒度允许的范围内尽可能均分；奇数筹码按一条固定的确定性规则
分配：**默认 — 给最接近按钮左侧的并列赢家。** 扑克手牌排名**没有花色优先级**（修正 RT-01 m3）；
基于花色的破平*仅*是一条房规（例如某些梭哈牌室），**默认关闭**，作为一个显式的规则集开关携带，
且绝不得在手牌评估（§5.3）内部实现 — 它是一项底池判给的破平，绝非手牌比较。这是一项确定性要求
（P2）：所有客户端都依规则集以相同方式判给奇数筹码。

## §5.6 摊牌与最小揭示

REQ-POKER-014 摊牌时，只有某个池的争夺者才揭示，且只揭示裁决所需的部分。上一位进攻者先亮；
其他人可依次亮牌或盖牌（muck）；若一名玩家无法赢得任何被争夺的池，他可以盖牌而不亮牌（标准的
"亮一张即亮全部，show one, show all"房规变体是一个规则集开关）。REQ-POKER-015 一手被揭示的牌
在能够获胜之前，先对照该玩家的隐藏牌承诺（§4.5）进行验证（§6.3 揭示脚本）。

## §5.7 结算与筹码移动

REQ-POKER-016 手牌结算依 §5.5/§5.6 将每个池路由给其赢家并更新筹码量。在链上，结算是一笔将底池/
筹注结构花费给赢家的交易（§6.3 结算脚本），或在微下注模式下，是一次通过 `Q*` 以整聪对账重新划分
锁定价值的 `BS` 通道更新（§2.2）。REQ-POKER-017 如果结算卡住（赢家不可达、某联署方缺席），恢复/
超时路径（§6.4）确定性地解决价值；资金绝不会被搁浅（P4）。

# §6 BSV 交易与 Script 模型

可执行的骨架。每个协议状态和合法后继都映射到可强制执行的价值移动和分支激活。每一笔交易都绑定到
它所花费的确切状态，使某个分支无法在不同的上下文下被重放。

## §6.1 交易类别

| Class | 用途 |
|---|---|
| Funding | 将玩家筹注锁入牌桌/底池结构；绑定 `gid` + `rulesetHash` |
| Commitment | 锚定熵承诺与各阶段洗牌承诺（§4.1、§4.4） |
| Deal | 将隐藏牌 UTXO 分配给座位（§4.5） |
| Action | 玩家在窗口内的一步行动（看牌/下注/跟注/加注） |
| Timeout | 决策截止后的默认分支（无玩家参与） |
| Reveal | 带证明地披露已承诺的牌材料（§4.6、§5.6） |
| Fold | 不揭示地交出隐藏手牌（§4.6） |
| Fair-play | 脚本内证明所承诺密钥与所用密钥相符（§4.7） |
| Settlement | 将底池路由给赢家（§5.7） |
| Recovery | 针对卡住对局的较长超时回退/退款（§6.4） |
| Table mgmt | TABLE_CREATE/JOIN/LOCK/ABORT/CLOSE |

（线缆名称是实现细节；这些是概念性类别。）

## §6.2 Genesis 升级后的 BSV Script 约束（关键 — P1，及一处修正）

REQ-TX-001 **`OP_CHECKLOCKTIMEVERIFY`（CLTV）和 `OP_CHECKSEQUENCEVERIFY`（CSV）在 Genesis 升级后
的 BSV 上是空操作（no-op）**，且绝不得用于强制时序。依赖它们的脚本什么也强制不了。这修正了任何
更早的/通用的设计（包括平台自身的架构系谱）将 CLTV/CSV 列入所需操作码的做法。REQ-TX-002 凡需要
时序之处（决策超时、恢复窗口、退款成熟期），在**交易层面**使用 `nLockTime` 和原始交易替换规则下
的输入 `nSequence` 字段来强制 — 这些机制作用于整笔交易，且在 Genesis 升级后仍然有意义。
REQ-TX-003 在适用之处结合经济性的保证金罚没激励（§2.2），交易层面的时序对协作/超时分支模型（P4）
而言已足够。REQ-TX-004 平台所用的操作码集合限于在 Genesis 升级后有意义的原语：签名检查
（`OP_CHECKSIG`/`OP_CHECKSIGVERIFY`/`OP_CHECKMULTISIG`）、哈希/相等
（`OP_SHA256`/`OP_HASH160`/`OP_HASH256`/`OP_EQUAL`/`OP_EQUALVERIFY`）、条件
（`OP_IF`/`OP_ELSE`/`OP_ENDIF`/`OP_VERIFY`），以及公平对局 EC 例程所需的数值/栈操作
（`OP_MUL`/`OP_ADD`/`OP_MOD`/`OP_DUP`/`OP_SWAP`/`OP_ROT`/
`OP_TOALTSTACK`/`OP_FROMALTSTACK`/`OP_2DUP`/`OP_NOTEQUAL` 等，如 GB2616862 第 55–60 页所示）。

## §6.3 分支绑定（防重放）

REQ-TX-005 每一笔交易至少绑定：`gid`（游戏 id）、`rulesetHash`、轮号、状态哈希、行动座位（在相关
之处）、经济状态、公共牌/明牌、隐藏承诺引用，以及一个后继状态承诺。这防止跨状态重放和过期分支的
滥用。跨牌状态转换携带的状态被逐字节检查（例如牌密文槽位的相等性，§4.5），使替换通不过脚本。

## §6.4 超时与恢复（交易层面）

REQ-TX-006 两层超时：一个短的**决策超时**（针对某个特定动作的不作为默认 — 对扑克而言，安全的
默认是：若看牌合法则看牌，否则弃牌/放弃；绝不强制下注，那会制造不对称的可被利用风险）和一个较长
的**恢复超时**（扣留揭示、结算卡住、失去法定人数），它从最后一个有效的已承诺状态解锁或重新分配
价值。REQ-TX-007 这些超时的"现在（Now）"取自链/中继锚定的高度/时间（§3.3 / §8），对于影响共识的
决策不取本地挂钟。REQ-TX-008 在对局开始前为以下情形签署一张**预签名回退图**：建桌中止退款、无
法定人数回退、建堆超时、发牌超时、动作超时默认、摊牌/揭示超时罚没、结算超时、闭桌退款。在相应
成熟期之后，价值经由一条已授权的路径回退 — 凭脚本图设计，而非善意。

## §6.5 承诺携带：OP_RETURN 还是 pushdata

REQ-TX-009 只需被锚定的死端承诺（洗牌阶段承诺、规则集哈希锚定）**可以**使用 `OP_RETURN`
（依 GB2616862）。REQ-TX-010 必须保留在花费链接的、可链式的交易图内部的承诺（例如经由 `VA` 的
审计/三式记账链接，§2.3）**必须**作为**活动脚本中的 pushdata 携带，而非 `OP_RETURN`**。这一区分
是功能性的，而非风格性的。

## §6.6 脚本模板（从 GB2616862 的 2 方演算示例放大而来）

平台需要以下模板族。GB2616862 给出 2 方/3 张的演算脚本；平台将它们参数化以适配 N 方和牌堆大小。
每个模板在 §19.C `[EXPAND: Part 3]` 中规范到字节排布层级；其结构：

- **Funding 模板** — 对玩家买入的 N-of-N（或门限）多签；绑定 `gid`+`rulesetHash`；配套退款（nLockTime 成熟期）已预签名。
- **逐牌选择/平仓锁** — 锁定到组合密钥 `Q_j` 的 UTXO（§4.3）；其花费是平仓/结算原语（而非发牌）。按当前活动签名模式签名（§4.3）：模式 A 在揭示时重建 `w_j` 并签名；模式 B 使用门限/多方签名且不重建完整密钥。
- **隐藏牌托管** — 分配给座位；弃牌交出路径；揭示路径；超时回收路径，全部以签名和 §6.3 的状态检查为条件（时序在 tx 层面，§6.2）。
- **揭示或超时（Reveal-or-timeout）** — 一个分支在成熟期之前接受一次有效的揭示开启（哈希开启检查 `H(face‖blind)=cmt`）；另一个分支在成熟期之后回退/罚没（nLockTime）。
- **Fold** — 证明对隐藏输出的控制，转移到死手，保留承诺（不披露牌面）。
- **底池/结算** — 在有效的获胜证明 + 签名下支付给赢家；或协作签名分配；或超时退款分支。
- **Fair-play** — 脚本内的洗牌点派生 + EC 点乘法证明（GB2616862 第 55–60 页）；仅当所承诺密钥能推导出所用密钥时才可赎回；否则被罚没。超出该专利 3 元素示例的放大是一个实测风险（REQ-CRYPTO-009）：单个 52 张 N 方脚本可能不切实际；回退方案是逐牌/逐批的公平对局交易，由 §19.C 的测量决定。
- **可选 2-of-3 TTP** — 需主动启用的配置，加入一个部分受信的第三签名方，仅用于争议时介入（GB2616862 §6.5.1）；默认关闭。

REQ-TX-011 每个模板都随附：一个肯定测试（解释器接受有效花费）、一组否定测试（每个应当失败的情形都
在解释器**内部**失败，P9），以及一个记录为可复现向量的实测线缆字节大小（P10）。

## §6.7 组合密钥 UTXO 的签名（依 §4.3 的模式决策）

REQ-TX-012 花费一个 `Q_j` 锁定的 UTXO 需要一个在 `Q_j` 下有效的签名，按当前活动签名模式（§4.3）
产生，经由可插拔的托管后端（§9.3）显式呈现：

- **模式 A（默认，第 1 阶段）：** 进行平仓的一方从揭示时披露的标量重建 `w_j = Σ_p s_{p,j}` 并签名。这以一个具体的、忠于专利的构造了结了第 1 阶段的签名问题 — 交付第 1 阶段不需要任何门限密码学。§4.3 的模式 A 后果（单局密钥、有界的手牌窗口暴露）适用，并在 §9.2 中强制执行。
- **模式 B（升级，第 2 阶段起）：** 一种无荷官的门限/多方 ECDSA（一个 `OB.custody` FROST/GG20 后端是候选）在不重建完整密钥的情况下产生签名。

两种模式下，输出签名都必须能在 `OP_CHECKSIG` 处通过验证，且在验证时与一个普通签名无法区分。
`[EXPAND: Part 2 — 模式 B 构造只在阅读 cardtable 的签名 API 之后才确定；记录的候选方案是无荷官的
门限 ECDSA。模式 A 不需要此类依赖，且已在此完整规范。]`

## §6.8 `[EXPAND: Part 3]` 完整的字节排布、每个模板的 sighash/`SIGHASH` 标志选择，以及完整的预签名回退图交易集合。

# §7 游戏状态机（框架 + 德州扑克参考实现）

## §7.1 FSM 框架

REQ-FSM-001 一个游戏模块实现 `GameModule`：
`init(ruleset, seats) -> GameState`、`getLegalActions(state, seat) -> Action[]`、
`apply(state, action) -> GameState`、`isTimeoutEligible(state, now) -> {seat, defaultAction}?`、
`isHandComplete(state) -> bool`、`settle(state) -> Payouts`，外加序列化。引擎（§3.3）驱动各模块；
模块中**不**含任何 I/O、网络、时间或随机性（P2）。每个状态节点都枚举其合法后继，包括超时默认（P4）。

## §7.2 德州扑克（第 1 阶段参考实现；先单挑）

一手牌的各阶段（每个都是一个已承诺状态；每个可操作阶段都有协作 + 超时出口）：下盲注 →
每座位发 2 张隐藏底牌 → 翻前下注 → 揭示翻牌（3 张公共牌）→ 翻后下注 → 揭示转牌（1 张公共牌）
→ 转牌下注 → 揭示河牌（1 张公共牌）→ 河牌下注 → 摊牌（最小揭示 §5.6）→ 结算（§5.7）→ 下一手
（按钮移动）。单挑的特殊规则（按钮下小盲、翻前先行动、翻后后行动）被编码进下注顺序中。
REQ-FSM-002 完整的德州扑克状态转换表 — 每个状态、触发、守卫、后继和超时默认 — 在 §19.E
`[EXPAND: Part 2]` 中规范；此处的散文流程对阶段顺序和两出口规则具有规范性。

## §7.3 其他变体（FSM）

所有变体都复用 §7.1 的 `GameModule` 接口和两出口规则（P4）。每个变体都按以下方式规范：其规则、
它所使用的**心智扑克原语**（明牌 vs 暗牌揭示、抽换操作），以及一张转换表或相对于德州扑克表
（§19.E）的显式增量。新需求延续 `REQ-FSM-*` 系列。

### §7.3.0 明牌、暗牌与抽换 — 三个新原语

REQ-FSM-003 一张**暗牌（down-card）**由 `draw(position)` 操作（§4.3）发出，然后**仅私下揭示给其
持有者**（§4.6 私有揭示）。一张**明牌（up-card）**（梭哈/razz）由同一个 `draw(position)` 操作发出，
然后通过一次 **N-of-N 协作揭示立即公开揭示**（§4.6 公开揭示，带 RT-01 M2 的超时默认）。
REQ-FSM-004 **抽换（draw）**操作（五张抽换）为：玩家将其隐藏牌 UTXO 中所选的一个子集交出到一个
死手状态而**不揭示它们**（一次部分弃牌，§4.6），并从洗好的牌堆中尚未发出的部分获发相同数量的全新
隐藏牌 UTXO（§4.3 `draw`）。被弃置的牌从不被揭示；替换牌仅私下揭示给抽换者本人。REQ-FSM-005 由于
梭哈/razz 暴露明牌，**下注顺序由暴露的牌面决定**，而非由按钮决定（见各变体）；这是引擎中一个不同
的排序函数，并被单独测试。

### §7.3.1 奥马哈（及底池限注奥马哈）

**规则。** 在结构上与德州扑克相同（§19.E）：盲注、四条下注街（翻前/翻牌/转牌/河牌）、一个按
3-1-1 揭示的共享五张公共牌。**差异：** 每座位获发**四**张隐藏底牌（而非两张），且摊牌时手牌由
**恰好两张底牌 + 恰好三张公共牌**组成（§5.3.2、REQ-POKER-005 — C(4,2)·C(5,3)=60 的评估，在
§19.D 中验证）。常见的下注结构是**底池限注**（PLO；§5.4），不过 NL/FL 奥马哈可按规则集选择（D3）。
REQ-FSM-006 奥马哈是带两处覆盖的德州扑克 FSM（§19.E）：(i) `DEAL_HOLE` 每座位抽取 **4** 张隐藏牌；
(ii) `SHOWDOWN` 使用奥马哈受约束的评估器。其他状态均不变。

**相对于 §19.E 的 FSM 增量（单挑；与德州扑克完全一样地推广至多方）：**

### §7.3.2 七张梭哈（高牌）

**规则。** 无公共牌、无盲注。每个玩家下一个**底注（ante）**。每个玩家在五个下注轮（"街"）中所发

- **第三街：** 两张暗牌 + 一张明牌（"门牌，door card"）。**最低的明牌**（按点数，平局时用一个**仅**用于 bring-in 选择的固定花色顺序打破，于规则集中声明 — 这是一个强制下注选择器，而非手牌评估的花色优先级，参见 RT-01 m3）下 **bring-in**。在小注级别下注。

- **第六街：** 一张明牌。

### §7.3.4 Razz（七张梭哈低牌）

**规则。** 七张梭哈**结构**（底注；第 3–7 街；两张暗牌、四张明牌、一张暗牌；七取五最佳），但
**最低**手牌按 **ace-to-five 低牌**取胜（§5.3.3、REQ-POKER-006；顺子和同花不计，A 为低，最佳是
轮 A-2-3-4-5 — 在 §19.D 中验证）。相对于梭哈高牌的顺序反转：

- **第三街 bring-in = 最高的明牌**（在低牌中高即坏；平局按同一声明的强制下注花色顺序，而非手牌评估的花色优先级）。
- **第 4–7 街：暴露的最低（最佳）牌面先行动。**

REQ-FSM-011 Razz 是带三处覆盖的 §7.3.2 梭哈 FSM：(i) bring-in 选择器 = 最高明牌；(ii) 第三街后的

规则（对低牌而言无意义）。8 玩家耗尽规则（REQ-FSM-008）以相同方式适用。

**相对于 §7.3.2（梭哈）的 FSM 增量：** 状态图 S0–S9 相同；按 REQ-FSM-011 替换 bring-in 选择器、

# §8 网络、发现、匹配撮合、防卡死

## §8.1 角色

REQ-NET-001 **中继**只作传输 + 索引；它绝不充当真相之源（P3）。它加速收敛、重广播、索引牌桌交易、
调度超时通知，并汇编记录誊本。真相是经验证的交易图，由每个客户端以相同方式重建（P2）。

## §8.2 两层模型

REQ-NET-002 Tier A（发现）：在线状态与地址 gossip — 第 1 阶段经由托管中继和一个 LAN 自动发现模式
（D4）；第 5 阶段加入一个 Bitcoin 式的 version/verack/getaddr/addr 对等层。Tier B（游戏对象传播）：
牌桌作用域的清单/对象中继（inv/getdata/object），带逐桌通道以实现快速收敛（Bitmessage 式），第 1
阶段经由中继的牌桌作用域扇出。

## §8.3 双路径传播

REQ-NET-003 每个动作同时发送 (a) 作为一笔真实交易发往节点/网络（规范路径：内存池存在性、双花可观察
性、最终确认）和 (b) 经由中继/P2P 直接发给牌桌对等方（速度路径：快速的本地状态收敛、对传播延迟的
容忍）。速度路径绝不覆盖规范路径；冲突按 §8.5 解决。

## §8.4 节点/链接口

REQ-NET-004 经由 `BS.node`（§2.2）：广播 tx；查询输出点/UTXO 状态；观察双花尝试状态；区块头/区块
查询；原始替换规则下的内存池冲突信号。平台跟踪其自身的牌桌级重广播和对等可见性，并能重新提交缺失
的交易（内存池内容可能变化；牌桌系统不单靠内存池记忆）。

## §8.5 确定性冲突解决

REQ-NET-005 对每个可操作阶段，恰好有一个可花费的阶段权（状态 UTXO）。同一阶段的两个相冲突动作不可
能都有效：被接受的后继是被下一个被接受的状态转换所引用的那个；较晚的相冲突尝试相对于被接受的路径
而言无效。REQ-NET-006 凡在阶段关闭之前需要一个确定性的本地破平之处，规则是固定的（默认：被一定门限
的对等方所引用的第一个有效交易；最终破平按字典序的 txid）。双花尝试因超时默认分支而在策略上变得
无用（缺席/冲突的行动者被走默认/弃牌；手牌继续进行）。

## §8.6 重连/恢复

REQ-NET-007 客户端可向某个对等方/中继请求记录誊本的缺口，并从有效交易集合确定性地重建当前状态；
恢复不改变任何关于真相的东西，只改变本客户端的视图（P2）。

## §8.7 `[EXPAND: Part 2/5]` 完整的中继线缆协议、在线状态/心跳、匹配撮合/大厅语义、NAT 穿透与互联网 P2P（第 5 阶段），以及叠加在密码学保证之上的反作弊启发式。

# §9 钱包与密钥管理

## §9.1 角色

钱包是一个游戏协议代理，而不仅仅是一个支付钱包：持有签名密钥；派生逐局/逐牌密钥；签署交易和链下
协议消息；在本地解密隐藏的牌；解读牌桌状态；呈现签名提示。

## §9.2 密钥层级

REQ-WALLET-001 托管边界内部每个玩家一个长期 secp256k1 密钥；逐局和逐牌标量通过绑定到 `(gid, j)`
的 HKDF 确定性派生（§4.2），因此设备只存储一个密钥，派生从 `(sk, gid, j)` 出发是确定性的且可审计，
并且旧局的密钥不泄露任何关于当前各局的信息。REQ-WALLET-002 为不同的工作使用不同的密钥集（签名 vs.
牌加密 vs. 会话），按构造实现最小权限。

## §9.3 托管边界（可插拔；TEE 可选）

REQ-WALLET-003 一个 `Custody` 接口抽象密钥存放于何处以及签名在何处发生：
`derive`、`sign`、`decryptToViewer`、`combineSignShare`。后端：（默认）软件托管，密钥持于钱包进程中，
在查看者路径之外绝不暴露给 UI/外壳；（升级）`OB.custody` 门限（FROST/GG20），使任何完整密钥都不
存在于某一处；（稍后，可选）一个硬件 TEE 后端 — 在不改变协议的前提下添加。没有任何阶段要求 TEE
（按方向指示）。REQ-WALLET-004 完整密钥处理遵循当前活动签名模式（§4.3）：在**模式 A**（第 1 阶段
默认）中，组合私钥 `w_j` 由进行平仓的一方在揭示时重建，因此托管后端必须将所有逐牌标量视为**单局**
秘密材料，绝不跨局复用（由对 `gid` 的 HKDF 绑定强制，REQ-WALLET-001），并且必须仅在有界的手牌窗口
内持有组合密钥锁定的资金；在**模式 B**中，门限后端在不重建完整密钥的情况下产生 `Q_j` 签名。托管
接口为模式 B 暴露 `combineSignShare`，为模式 A 暴露 `reconstructAndSign`（受限作用域、单局、经审计）。

## §9.4 微下注托管

REQ-WALLET-005 在微下注模式下，通道密钥和 1 聪保证金经由 `BS`（§2.2）管理；钱包将通道的
open/transfer/close/contested 呈现为带显式提示的签名动作。

# §13 工程标准、需求登记册、可追溯性

## §13.1 标准

REQ-ENG-001 本项目按 **NASA NPR 7150.2** 软件保障实践，以及**适配于垃圾回收运行时的 JPL "Power of
Ten" 规则改编版 — 而非字面合规（修正 RT-01 M5）**来构建。原始规则面向嵌入式安全关键场景中的 C；
在 TypeScript/Go 中映射如下：

- **原样采纳：** 将控制流限制为简单结构 / 共识与脚本路径中无递归；为共识路径中的每个循环设定一个固定的、可证明的上界；在引擎边界使用运行时断言（契约式设计）；检查每个返回/错误值；保持函数小且单一职责；将所有警告视为错误（`tsc --strict`、Go `vet` + 等价于 `-Werror`）。
- **改编：** "限制数据作用域" → 核心中无无类型的 `any`、带编译期穷尽性的 `switch` 穷尽处理、严格空值检查；"限制预处理器" → 限制类宏元编程/代码生成，并为任何残留者编写文档。
- **在 GC 运行时中不适用（明示，而非悄然舍弃）：** 规则 3（初始化后不进行动态内存分配）以及裸指针/别名限制在垃圾回收下没有字面意义。取而代之，核心在热共识路径中使用有界的、预先定尺寸的工作结构，并避免在该路径内部进行无界分配；这是一种*改编*，本文档不声称符合规则 3。

因此其标题口径是"NPR 7150.2 保障实践 + 一份有文档记载的 Power-of-Ten 改编"，绝非"Power-of-Ten 合规"。

## §13.2 需求登记册

REQ-ENG-002 本文档中的每个 `REQ-*` ID 都是一个机器可读登记册（`/spec/requirements.yaml`）中的一条
条目，包含：id、文本、所属章节、所属模块、满足它的源文件、满足它的测试、状态。该登记册在 CI 中
生成/校验。

## §13.3 可追溯性矩阵

REQ-ENG-003 一个需求可追溯性矩阵将每条需求 → 代码 → 测试映射起来。如果任何需求没有满足它的测试，
或任何共识路径源文件未被追溯到某条需求，则 CI 失败。（镜照 `overlay-broadcast` 的实践，INV-OB-3。）

## §13.4 边界落入源码

REQ-ENG-004 每条所声明的边界/限制（P7/P8）都在源码中被断言 — 例如审计边界（INV-VA-2）、N-of-N 活性
限制（§4.8）、regtest 默认（§10.5） — 使任何未来的变更都不会悄然掩盖它。

## §13.5 `[EXPAND: Part 3]` 从本文档中每个 `REQ-*` 播种生成的完整需求登记册，含负责人和测试 ID。

记录誊本。

## §14.5 可复现向量（P10）

REQ-TEST-005 `reproduce` 重新生成每个已提交的向量（手牌评估结果、底池分配、脚本线缆字节大小、记录

## §14.6 对抗性套件

REQ-TEST-006 在每个阶段断线；过期/重复动作；超时竞争；扣留揭示；相冲突的花费；内存池逐出 + 重新

# §15 模块 API / SDK 契约

# §18 威胁模型

每个威胁都列出**限定它的机制**和一个**诚实的残余风险**（P8）。"受限"绝不意味着"被消除"。威胁
被分组；ID 为 `THR-*`。

## §18.1 经济性威胁

- **THR-ECON-1 过期状态广播（发布一个更旧、更有利的通道/牌桌状态）。** 限定：在微下注中，过期广播会使固定的一聪保证金被罚没给诚实方（§2.2、INV-BS-2），因此作弊的期望值为负；在链上牌桌结算中，一个被取代的状态无法花费当前状态的 UTXO（§8.5）。残余：一个把捣乱看得比一聪更值的攻击者仍可烧掉其保证金来扰乱；保证金大小是一个可调参数，而非威慑的证明。
- **THR-ECON-2 结算操纵（主张一个并未赢得的底池）。** 限定：结算花费需要一个对照隐藏牌承诺验证过的有效获胜证明（§5.6、§6.6 结算模板）和签名；未经证明的主张在解释器内部失败（P9）。残余：取决于揭示/承诺绑定的健全性（§4.5–§4.6），以及在模式 A 下取决于有界窗口密钥论证（§4.3）。
- **THR-ECON-3 输牌后拒绝结算（扣留一个所需的签名 / 标量）。** 限定：预签名回退图和恢复超时（§6.4）在成熟期之后经由一条已授权路径移动价值；输家无法搁浅底池。残余：诚实赢家被施加一个恢复延迟（超时窗口）；该窗口是一项 UX/安全权衡。
- **THR-ECON-4 捣乱 / 慢摇（仅在截止时刻行动以浪费时间）。** 限定：带不作为默认的决策超时（§6.4）为每个动作的延迟设上限；惯犯可被中继的大厅声誉降权（§8.7，稍后）。残余：在超时预算内，攻击者仍可最大化延迟；只能靠调节超时来缓解。

## §18.2 公平性威胁

- **THR-FAIR-1 洗牌偏置（扭曲牌堆顺序）。** 限定：可验证的分布式洗牌复合了每一方的秘密置换，因此没有任何单方能固定顺序（INV-CT-1）；承诺-揭示熵（§4.1）阻止晚到的熵选择。残余：所有各方的*完整*联盟可以商定顺序 — 但那样就没有诚实玩家可被欺诈了；安全性随联盟规模优雅地降级，直至 N−1。
- **THR-FAIR-2 做记号的牌 / 不匹配的密钥（使用与所承诺不符的密钥）。** 限定：公平对局交易在脚本内证明所承诺密钥能推导出所用密钥；不匹配将丧失被绑定的资金（§4.7）。残余：取决于公平对局脚本正确，且其放大方案（REQ-CRYPTO-009 / RT-01 M3）已就位；在 §19.C 测量它之前，此控制项已被规范但未在牌堆规模上得到证明。
- **THR-FAIR-3 平方根分支泄露（将密钥搜索空间减半）。** 限定：固定一个分支并保持该选择一致（REQ-CRYPTO-007）；资金短命（§4.3 模式 A 窗口）。残余：若分支选择泄露则减小为原来的 1/2；在手牌窗口内仍计算上不可行，但如实声明，而非搪塞。
- **THR-FAIR-4 揭示时合谋（模式 A 标量复用）。** 限定：逐牌标量是单局的且绝不复用（REQ-WALLET-004）；组合密钥资金只在该手牌期间存活。残余：在单局之内，合谋玩家本就看到彼此已揭示的牌（这是任何心智扑克联盟固有的）；对需要的人，模式 B 完全消除了完整密钥重建的暴露。

## §18.3 活性威胁

- **THR-LIVE-1 扣留揭示（玩家或公共牌）。** 限定：每次揭示 — 包括每条公共牌街 — 都是一次带超时默认的 N-of-N 协作转换（§4.6、RT-01 M2）；恢复解决价值，已配置之处适用保证金罚没。残余：手牌在走默认之前会卡住超时窗口那么长；一个铁了心的扣留者把一次大概率的输牌转化为一次被迫超时，绝非一次盗窃。
- **THR-LIVE-2 法定人数掉线（某方在手牌进行期间断线）。** 限定：超时默认/恢复路径（§6.4）；不尝试任何部分重洗（REQ-CRYPTO-012）。残余：手牌通过回退被作废/结算，而非用替补继续；牌桌以当时就座的集合进行到下一手（§4.9）。
- **THR-LIVE-3 内存池逐出 / 不传播。** 限定：平台跟踪牌桌级重广播并能重新提交；它不单靠内存池记忆（§8.4）。残余：传播延迟会减慢收敛；速度路径（§8.3）缓解的是 UX 而非底层网络。

## §18.4 客户端与密钥威胁

- **THR-CLIENT-1 从钱包/设备提取密钥。** 限定：托管边界（§9.3）；模式 B 门限托管使任何完整密钥都不存在于某一处；稍后可选 TEE 后端。残余：软件托管（第 1 阶段默认）信任本地设备；设备被攻陷即该玩家的密钥被攻陷 — 坦白声明；平台不声称第 1 阶段具备抗设备攻陷能力。
- **THR-CLIENT-2 手牌视图泄露（屏幕捕获、读取已解密牌面的恶意软件）。** 限定：解密只经由受控的查看者路径（§11.5）。残余：被攻陷的客户端能泄露其自己玩家的视图；这在协议层面无法防止，仅作为部分缓解来声称（屏幕加固是一条可选的后续轨道）。
- **THR-CLIENT-3 恶意中继。** 限定：中继只作传输/索引，绝不充当真相之源（P3、§8.1）；客户端独立推导状态，并通过从有效交易集合重建来检测一个撒谎的中继。残余：恶意中继能降低活性（丢弃/延迟消息）并尝试日蚀（eclipse）；由规范网络路径（§8.3）缓解，并在第 5 阶段由对等层（§8.7）缓解。

## §18.5 协议威胁

- **THR-PROTO-1 将某分支重放到不同状态上。** 限定：每一笔交易绑定 `gid`、`rulesetHash`、轮号、状态哈希和后继承诺（§6.3）；被重放的分支在解释器中通不过绑定检查。残余：除哈希绑定的健全性之外别无残余。
- **THR-PROTO-2 双花 / 相冲突动作。** 限定：每阶段一个可花费的阶段权；相冲突的尝试不可能都有效；确定性冲突规则（§8.5）和超时默认使双花在策略上变得无用。残余：确认前的排序分歧，由 §8.5 解决；仅在确认时才最终确定。
- **THR-PROTO-3 未确认牌桌交易的重组。** 限定：对离开牌桌的价值，玩法将确认视为最终结算；手牌进行中的状态从已商定排序重新推导，并受恢复路径（§6.4）保护。残余：一次深度重组可能使一次未确认结算失效；已离开牌桌的价值会等待确认深度（一个规则集参数）。

## §18.6 完整的信任面（P8 — 除此之外没有任何东西是"无信任的"）

一个运行未被攻陷客户端的正确、诚实玩家所信任的恰好是：

1. **secp256k1 上的离散对数 / DDH 难度**（洗牌、密钥、签名）。
2. **协议参与者对未在链上强制执行的步骤的正确链下执行**（链上部分由解释器强制；链下编排被信任为遵循规范，凡触及链状态之处，偏离会被公平对局、承诺和重放捕获 — 但从不触及链的纯链下不当行为只受超时/经济机制限定）。
3. **各原语的健全性：** HKDF、AEAD（AES-256-GCM）、SHA-256/双 SHA-256、ECDSA，以及（模式 B）门限方案。
4. **节点与 BSV 共识完整性：** 本地/已连接节点强制执行 Genesis 升级后共识，且工作量证明保护区块头链。
5. **所配置层级的托管后端完整性：** 软件（设备信任）、门限（无完整密钥），或 TEE（硬件飞地） — 取决于所配置的那种（§9.3）。

在此清单之外，平台不要求信任。"无信任（trustless）"一词在任何地方都不被无条件使用；凡散文中使用
"非托管"或类似措辞之处，指的是所强制的特定属性（没有运营方持有牌堆或决定结果），由这个面所限定。

## §19.B 边池算法（附完整演算示例）

**输入：** `contrib[p]` = 玩家 `p` 本手牌投入的总筹码（包括已弃牌玩家 — 他们的筹码在底池中但他们
不能赢）；`live` = 未弃牌的玩家集合（有资格获胜）。

**算法（确定性；满足 P2）：**

1. 构建跨**全部**玩家的、各不相同的正投入层级的有序集合：`L₁ < L₂ < … < L_m`。
2. `prev ← 0`。对 `i = 1..m`：
   a. `increment ← Lᵢ − prev`。
   b. `contributors_i ←` 所有满足 `contrib[p] ≥ Lᵢ` 的玩家（在场**或**已弃牌）。
   c. `potᵢ.amount ← increment × |contributors_i|`。
   d. `potᵢ.eligible ←` 满足 `contrib[p] ≥ Lᵢ` 的**在场**玩家。
   e. `prev ← Lᵢ`。
3. 任何满足 `|potᵢ.eligible| = 1` 的 `potᵢ` 都**无须摊牌直接判给该玩家**（这是"未被跟注/退回"情形 — 唯一的有资格出资者将其取回）。
4. 摊牌时，按 `compare`（§5.3）将每个剩余的 `potᵢ` 判给有资格的最佳**在场**手牌；平局按 §5.5.1 分配（均分；奇数筹码按确定性规则，绝不按花色）。
5. **守恒检查（断言，REQ-ENG）：** `Σ potᵢ.amount = Σ_p contrib[p]`。违反即是缺陷。

**完整演算示例（3 人桌，翻前多人全下，为清晰起见无弃牌）。**

筹码/投入：`A = 100`、`B = 60`、`C = 40`。三人全下。`live = {A,B,C}`。

- 各不相同的层级：`L₁=40, L₂=60, L₃=100`。
- **池 1（主池）：** 增量 `40−0=40`；满足 `contrib ≥ 40` 的出资者 = {A,B,C}（3）；金额 `40×3 = 120`；有资格（在场）= {A,B,C}。
- **池 2（边池 1）：** 增量 `60−40=20`；满足 `contrib ≥ 60` 的出资者 = {A,B}（2）；金额 `20×2 = 40`；有资格 = {A,B}。
- **池 3（边池 2）：** 增量 `100−60=40`；满足 `contrib ≥ 100` 的出资者 = {A}（1）；金额 `40×1 = 40`；有资格 = {A} → **唯一有资格 ⇒ 退回给 A**（步骤 3）。
- 守恒：`120 + 40 + 40 = 200 = 100+60+40`。✓

假设手牌强度 `C > B > A`：

- 主池 120 → **C**（{A,B,C} 中最佳）。C 净额：`−40 + 120 = +80`。
- 边池 1（40）→ **B**（{A,B} 中最佳；C 无资格）。B 净额：`−60 + 40 = −20`。
- 边池 2（40）→ **退回给 A**。A 净额：`−100 + 40 = −60`。
- 净额之和：`+80 −20 −60 = 0`。✓（C，最佳手牌，赢取的不可能超过 C 出资过的池 — C 被封顶在主池，恰如规则所要求。）

**实现必须覆盖的边界情形（每个都是一个 §19.D 向量）：** 一名已弃牌玩家，其投入位于一个他无法赢的
池中；两名玩家在某边池上并列（均分 + 奇数筹码规则）；一次恰好等于另一玩家筹码量的全下（层级重合
→ 一层，而非两层）；一次未被跟注的最后加注（唯一有资格的顶层池被退回）。
