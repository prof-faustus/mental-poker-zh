## 2. 硬性规则 —— 请勿违反（委托人的评审框架）
- **仅限 BSV。** 后 Genesis 的 BSV。没有 BTC 代码，没有 BTC 假设。BTC = 自动失败。
- **OP_RETURN 被禁止。** 绝对的。任何 OP_RETURN = 自动拒绝（见 §3，第一号修复）。
- **零捏造。** 每个数字都追溯到一个来源或一个声明的 `TRACKED ASSUMPTION`。通过运行代码生成数值（如 §19.D 所做），绝不凭记忆写出。
- **无假设 / 无隐藏假设。** 声明每一个设计选择；在未确定之处标注 `DECISION REQUIRED`。隐藏假设是最糟糕的缺陷类别。
- **不夸大。** 只陈述该构造所保证的内容；枚举信任面。
- **禁止道歉。** 直接、简洁。坦率承认错误，修复它们，继续前进。
- **工程标准：** NPR 7150.2 + 一份*有文档记录的 Power-of-Ten 适配*（不是字面合规 —— 规则 3/指针在 GC 中为 N/A）、需求可追溯性矩阵、通过**真实 Script 解释器**的测试（否定测试在其*内部*失败；不做签名抽查）、可复现的已提交向量。
- **不灌水。** 一份密集的 10k 行规格说明胜过灌水的 50k 行。目标是 10k–50k 行的*真实*内容，通过多轮迭代达成。

## 4. 需保留的已核验技术事实（不要错误地重新推导）
- **CLTV/CSV（`OP_CHECKLOCKTIMEVERIFY`/`OP_CHECKSEQUENCEVERIFY`）在后 Genesis 的 BSV 上是 NO-OPS。** 所有计时都在**交易层级**（在原始替换规则下使用 nLockTime + nSequence）。绝不在脚本内进行。（§6.2，REQ-TX-001/002。）
- **GB2616862A**（"Set shuffling"，Burns & Wright；申请人 nChain Licensing AG；引用为 **A**，而非 B —— 授权未确认）是洗牌/结算原语：洗牌密钥 `P'=(s,±√(s³+7))`，私钥 = x 坐标，公钥 `P=s·G`；每张牌的组合密钥 `Q_j=Σ P` 通过点加法得到；两轮加密（先共享标量，然后逐元素的不同标量）+ 重排 = 洗牌，在揭示时通过逆标量反转；结算通过 2-of-2 出资多签 + nLockTime 退款 + 分级下注交易 + 公平博弈脚本 + 可选的 2-of-3 TTP。已给出的示例是 2 方/3 牌的最高牌获胜 —— **扑克是向位置发牌**（规格说明的 `minted→drawn→revealed|folded→discarded` 生命周期；§4.3），而组合密钥的花费是**收尾结算，而非发牌**。
- **签名模式（D9，§4.3）：** 模式 A（专利字面：在揭示时重建 `w_j`；单局密钥；有界手牌窗口；**阶段 1 默认**）对比 模式 B（门限/无重建；升级版）。绝不在交付模式 A 的同时声称模式 B 的"无完整密钥"属性。
- **依赖栈（接受为已存在，Claude Code 拥有它们）：** `cardtable`（mental-poker 基底；部分 In-Between；协议已完整规格化，可离线/仿真运行 —— 并非那五种扑克变体，那是一份 Grok 文档的夸大），`bonded-subsat-channel`（亚聪微支付 + 一个**自包含的嵌入式 BSV 节点**，被复用作本地 regtest 后端），`verifiable-accounting` ×3（Merkle 锚定的选择性披露审计；用 pushdata 而非 OP_RETURN），`overlay-broadcast`（密钥图广播加密，吊销 = 未花费的过期输出，FROST/GG20 门限托管）。`revocable-nft-tee` / `triple-entry-bsv-sql` 曾在 Grok 文档中，但**不在**作者发布的列表里 —— 将 NFT/吊销视为构建于 `overlay-broadcast` 之上的、较后的一条线索。
- TEE 是**可选的**（托管后端可插拔；没有任何阶段需要它）。
