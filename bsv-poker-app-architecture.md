### 本文档的来源与禁捏造规则
本文档断言**任何已测量或已计算的值都没有**。一份关键任务规范通常会承载的每个量 —— 延迟预算、字节大小、吞吐量、内存上限、
覆盖率百分比 —— 在此要么作为明确标记为
`TRACKED ASSUMPTION` 的**设计目标**出现（构建必须达到并**测量**的目标，而非已达到的声称），
要么作为 `DECISION REQUIRED` 出现。凡需产出某个值之处（手牌评估向量、脚本
字节排程、性能数字），本文档说明**任务、方法与验收**，
并将其指派给 Claude Code（§A19）。作者出于任何原因都不运行代码。

### 合规用语
关键词 **MUST**、**MUST NOT**、**REQUIRED**、**SHALL**、**SHALL NOT**、**SHOULD**、
**SHOULD NOT**、**MAY** 的用法依照 RFC 2119/8174。**缺陷（defect）**是对
**MUST**/**SHALL** 的任何偏离。**阻断项（blocker）**是使某阶段无法交付的缺陷。本文档中的需求 ID
使用前缀 `REQ-APP-*`，并在 §A18 中登记；每条都映射到一个归属
模块（§A2–§A8）、一种验证方法（§A16）、以及一道验收闸门（§A18.3）。该登记册
与核心规范的登记册（核心 §13.2/§19.F）保持步调一致 —— 二者从不
重复定义同一需求；某条精化了核心需求的应用需求会引用它。

# §A1 应用概览

## §A1.1 "运行"的含义
"运行"是一个进程，它（REQ-APP-001）将密钥保存在托管边界之后，且仅通过引擎推导牌桌
状态；（REQ-APP-002）将每个玩家动作在**两条路径上
同时**发送 —— 作为真实 BSV 交易发往网络（canonical），并经由 relay 直接发往
牌桌对等方（speed）—— 依照核心 §8.3；并且（REQ-APP-003）渲染牌桌，并为每笔交易获得
明确、知情的签名同意（不静默签名；核心 §11.6）。两个
程序的差异仅在于其下层服务如何运行以及如何持久化。

## §A1.2 两个程序并列对比

| Concern | Windows 桌面（Tauri） | Web 客户端 |
|---|---|---|
| 服务（node、relay、indexer） | 内嵌，**由应用监管**（§A3） | 远程：捆绑本地（dev/regtest）或托管（internet）（§A4） |
| 持久化 | SQLite（AD9） | IndexedDB（AD10） |
| 链访问 | 本地 node（`BS.node`）+ 需要处用 SPV | SPV 边缘验证 + relay/indexer 投影 |
| 托管隔离 | 受信进程（Rust 侧或隔离 worker） | 隔离 worker；密钥从不进入 DOM |
| "无需外部服务即可游玩" | 内嵌 node+relay（D5/D6） | 捆绑本地模式，或一个托管的 relay+node |
| 更新 | 已签名安装器；自动更新稍后（§A14） | 重新部署包 |

## §A1.3 本层必须信守的产品承诺（核心 §1、§17、D1）
用户在 Windows 上安装或打开 web 应用；创建或导入钱包；找到其他玩家；
在 regtest 上坐入一张单挑 NL Hold'em 牌桌；打完一手完整手牌 —— 熵 commit/reveal、
分布式洗牌、加密发牌、各下注轮、**弃牌而不亮出手牌**、
最小揭示摊牌、确定性结算 —— 全程伴随决策与恢复超时；
并能导出并确定性地重放 transcript。§A10 的完成定义
使这一切可核查。

## §A2.2 确定性边界（REQ-APP-010）
UI 与 app-services MUST 将引擎视为纯函数（核心 §3.3、REQ-ARCH-001/002）：
任何 UI 代码都不计算合法性、结果、底池拆分或手牌排名；它调用
`getLegalActions`/`deriveState`/`validateTransition`（核心 §15.2）并渲染结果。"现在"
仅作为一个由链/relay 锚定的高度/时间推导出的显式参数进入（核心 §3.3、
§6.4），任何影响共识的决策都绝不取自 `Date.now()`（REQ-APP-011）。正是这道边界
使确定性重放（§A16.4）与跨客户端一致性可被测试。

## §A2.3 app-services（客户端服务层，TypeScript）
位于 `ui-core` 与 SDK 之间的一个轻薄层，在两个外壳间完全一致：
- **连接管理器**（§A7）：拥有 canonical 路径（通往 node/网络）与 speed 路径（经由 relay
  通往牌桌对等方）；暴露一个向两者扇出的统一 send API（核心 §8.3）；管理
  presence、牌桌频道订阅、重连。
- **同步/状态推导器**（§A2.2）：从 indexer 投影 +
  对等方馈送 + 本地持有的 commit/reveal 材料组装有序的 valid-tx 视图；调用引擎；将状态发射给
  view-model。
- **托管客户端**（§A8）：通往 `Custody` 接口（核心 §9.3）的进程内代理；请求
  `derive`/`sign`/`decryptToViewer`/`combineSignShare`；从不在 UI 内存中持有原始密钥。
- **超时/观察者**（§A11）：依据锚定的高度/时间计算超时资格；呈现
  后果状态；除非身为执行中的客户端角色，否则绝不自行推进默认动作
  （推进是任何对等方都可发起的交易，核心 §6.4）。
- **持久化网关**（§A13）：通往 SQLite/IndexedDB 的唯一有类型之门；所有读/写均
  经由它；强制执行模式与脱敏规则。

## §A2.4 模块/包布局（契合核心 §16）

`/packages/sdk`（核心）、`/apps/client-web`、`/apps/client-desktop`、`/apps/relay-go`、
`/apps/indexer-go`、`/vm`。应用层新增 `ui-core` 与 `app-services`；其余皆为
消费。（REQ-APP-012：任何应用包都不得直接 import 某个依赖仓库；它须经由
SDK/适配器，核心 §2.6。）

# §A3 桌面运行时与监管（Tauri）

桌面程序是一个 Tauri 应用：一个 **Rust 监管器**（Tauri main）加上一个运行 `ui-core` 的
**WebView2**。监管器之所以存在，是为了让非技术用户双击即可游玩。

## §A3.1 进程模型
```
Tauri main (Rust 监管器)
  ├─ 派生并监管内嵌 BSV node (regtest, BS.node, D6)
  ├─ 派生并监管 relay-go
  ├─ 派生并监管 indexer-go
  ├─ 健康检查 · 有序启停 · 崩溃重启
  ├─ 端口分配 · 数据目录管理
  ├─ 托管受信操作（密钥从不离开此边界）
  └─ IPC 桥  ⇄  WebView2 (ui-core)
WebView2 (ui-core, TS/React)
  ├─ 全部 UI、引擎状态推导、签名提示
  ├─ 经由托管边界进行本地牌解密（仅渲染牌面）
  └─ relay/对等方连接（Tier A 发现，Tier B 牌桌频道）
```

## §A3.2 服务生命周期状态机（REQ-APP-020）
监管器 MUST 实现此生命周期，并将其作为状态流暴露给 UI：

| State | Entry | Exit（成功） | Exit（失败） |
|---|---|---|---|
| `INIT` | 应用启动 | 配置加载完成 → `START_NODE` | 配置无效 → `FATAL` |
| `START_NODE` | 来自 INIT | node 健康 OK → `START_INDEXER` | node 失败 N 次重试 → `DEGRADED(node)` |
| `START_INDEXER` | node 健康 | indexer 健康 OK → `START_RELAY` | indexer 失败 → `DEGRADED(indexer)` |
| `START_RELAY` | indexer 健康 | relay 健康 OK → `READY` | relay 失败 → `DEGRADED(relay)` |
| `READY` | 全部健康 | （稳态） | 任一服务死亡 → `RECOVER(service)` |
| `RECOVER(svc)` | 某服务已死亡 | 重启 OK → `READY` | 重启超出策略 → `DEGRADED(svc)` |
| `DEGRADED(svc)` | 反复失败 | 用户/手动重试 → `RECOVER(svc)` | — |
| `SHUTDOWN` | 退出 | 反序停止 → `EXIT` | 超时后强制终止 → `EXIT` |
| `FATAL` | 不可恢复 | — | 呈现诊断信息，无法游玩 |

REQ-APP-021：有序启动（node → indexer → relay）；反序关闭。REQ-APP-022：
重启策略有界（最大尝试次数、退避）—— 两者皆为构建设置并呈现的 `TRACKED ASSUMPTION`
值；绝不无界重启循环（Power-of-Ten 有界循环纪律，§A17）。
REQ-APP-023：`READY` 是大厅允许创建/加入牌桌的唯一状态；其他
状态显示启动/健康屏幕（§A6.1）。

## §A3.3 IPC 契约（Tauri main ⇄ ui-core）—— *接缝，非代码*（REQ-APP-024）
命令/事件族；确切签名归 Claude Code，按核心 §15 与
`protocol-types` 定型。完整目录见 **Appendix I**。
- `services.*` —— `start`、`stop`、`status`（§A3.2 生命周期的事件流，按服务）。
- `chain.*` —— 透传至 `Chain`（核心 §15.9）：`broadcast`、`outpointStatus`、`txStatus`、
  `headers`、`conflicts` —— 使 UI 无论 node 是本地还是远程都使用同一 SDK 表面。
- `custody.*` —— `derive`、`sign`、`decryptToViewer`、`combineSignShare`（核心 §15.6、§9.3）：仅返回
  结果；原始密钥材料 MUST NOT 穿越 IPC 边界（REQ-APP-025）。
- `config.*` —— `runtime`（端口、网络、数据目录、标志）、`setNetwork`（受主网
  标志守护，§A3.5）。
- `diag.*` —— `logs`、`metrics`、`bundleDiagnostics`（已脱敏，§A12.5）。

REQ-APP-026：每条 IPC 消息在两侧都依其模式校验；无法识别或
畸形的消息被拒绝并记录，绝不部分应用（fail-closed，§A11）。

## §A3.4 端口、数据目录、regtest 默认值
REQ-APP-027：端口被分配并写入 UI 读取的运行时配置；无硬编码端口。
REQ-APP-028：SQLite 存储与 node 的区块/UTXO 存储位于一个按用户划分的数据目录之下，
路径有据可查（§A13、§A15 runbook）。REQ-APP-029：内嵌 node 默认为 **regtest**；
网络是数据目录命名空间的一部分，故 regtest 与（受标志的）主网从不共享状态。

## §A3.5 主网标志（REQ-APP-030；核心 §10.5、REQ-VM-007）
主网仅在一个明确命名的研究代码标志之后可达。监管器 MUST 拒绝
任何非 regtest 网络，除非该标志已设；只要该标志生效，UI MUST 显示一条无法忽视的横幅；
处处默认为 regtest。未设标志绝不触碰真实资金。

## §A3.6 桌面失败模式（交叉引用 §A11）
- node 无法启动（端口冲突、存储损坏）→ `DEGRADED(node)`、诊断、引导式恢复
  （在明确确认后重新初始化 regtest 存储）。
- relay/indexer 在手牌中途崩溃 → `RECOVER`；UI 保留牌桌视图、标记连接性
  降级，并依赖 canonical 路径 + 重连（§A7.6）；不臆造任何牌桌状态。
- WebView 崩溃 → 监管器重启 WebView，UI 从持久化 +
  transcript 缺口（§A7.6）重新推导状态；真相不丢失（它存于 tx 图中）。

# §A4 Web 运行时

web 客户端是由 Vite 构建、作为静态包提供的同一份 `ui-core`。

## §A4.1 连接模式（REQ-APP-040；核心 §8、D4）
- **捆绑本地**（开发者/regtest）：一个本地 relay+node 伴随件 —— 与桌面内嵌的同一
  node —— 经由 loopback 访问。
- **托管**（internet 游玩）：一个已部署的 relay+node。
连接管理器（§A7）抽象掉使用的是哪种；切换模式 MUST 不改动任何 UI 代码
（REQ-APP-041）。每个环境的默认值为 `AD-OPEN-1`（DECISION REQUIRED，§A20）。

## §A4.2 持久化（REQ-APP-042；核心 §11.2）
IndexedDB 用于密钥（已封装，§A10）、牌桌状态、transcript。`localStorage`/`sessionStorage` MUST
NOT 持有承重状态。模式与迁移见 §A13。

## §A4.3 链访问（REQ-APP-043；核心 §8.4、§34）
钱包处的 SPV 边缘验证：用于终局相关检查的 headers + Merkle 证明；relay/
indexer 投影用于牌桌状态；二者皆不被信为真相之源（核心 P3）。

## §A4.4 浏览器托管（REQ-APP-044；核心 §9.3、§11.5）
一个隔离 worker（Web Worker / WASM 模块）持有密钥并执行 sign/derive/decrypt；
渲染出的牌面仅存于受控的查看器路径中；密钥从不进入 DOM。后端
可插拔（软件默认；经由 `OB.custody` 的门限；TEE 稍后）。

## §A4.5 Service worker / 离线（AD-OPEN-2，DECISION REQUIRED，§A20）
service worker 是否缓存包以实现弹性重连，是一项已声明的未决决策；若
采用，则 MUST NOT 缓存密钥材料或牌桌机密。

## §A4.6 Web 失败模式（交叉引用 §A11）
- 与 relay/node 失联 → 带退避地重连；UI 显示连接性降级并
  在重连时重新推导（§A7.6）。
- IndexedDB 配额/驱逐 → transcript 与密钥可重新获取/重新推导；客户端发出警告
  并重新同步；承重状态不静默丢失（REQ-APP-045）。
- 手牌中途关闭/刷新标签页 → 返回时，从持久化 + transcript 缺口重建（§A7.6）。

# §A5 共享 UI 核心（一个核心，两个外壳）

一份 TypeScript/React 核心（核心 §11.1、§3.2），包 `/packages/ui-core`，由两个外壳消费。

## §A5.1 状态管理（REQ-APP-050）
单一的单向 store。输入：引擎推导的 `tableState`（§A2.2）、连接状态、
托管/签名状态、持久化。store **不**持有业务逻辑；reducer 仅将
SDK/引擎输出投影为渲染状态。无直接突变；UI 中不计算任何派生的游戏真相。

## §A5.2 View-model 契约（REQ-APP-051）
每个屏幕（§A6）一个 view-model，每个都是纯投影 `(engineState, connStatus, custodyStatus,
config) → ScreenProps`。view-model 在隔离中做单元测试（§A16.2）。组件是
展示性的，除局部 UI 瞬态外无状态。

## §A5.3 组件清单（REQ-APP-052）
已编目，各自带 props、状态（loading/empty/error/ready）、以及一项测试义务（Appendix
IV）：大厅列表、建桌表单、座位环、Board/明牌行、底池/边池显示、动作栏
（看牌/下注/跟注/加注/弃牌/换牌/停牌）、下注计量器（经由下注接口感知 NL/PL/FL，核心
§5.4）、计时器 + 后果横幅、手牌查看器（绑定托管）、签名模态框、摊牌面板、
结算摘要、重连/恢复横幅、transcript/重放查看器、主网横幅。

## §A5.4 交互与渲染规则（REQ-APP-053）
仅显式处理器（`onClick`/`onChange`）；**不用** HTML `<form>` 提交（核心 §11.3、
REQ-UI-003）。每个屏幕都定义其 loading/empty/error 状态。React 错误边界包裹每个
屏幕，使一处渲染故障只降级一个屏幕，而非整个应用（§A11）。事件处理器中
不做任何业务决策；处理器调用 app-services/SDK 并渲染结果。

## §A5.5 无障碍与国际化（REQ-APP-054；在 §A6 中按屏幕展开）
键盘可操作的控件、所有交互元素上的 ARIA 角色/标签、可见焦点、足够的
对比度、不靠颜色单独传达信息（花色也带有一个字形/字母 —— 且绝不
暗示花色优先级，核心 §5.5.1）、针对回合/计时器/后果变化的屏幕阅读器
播报、以及用于 i18n 的字符串外化层（英语优先）。无障碍是一项经测试的
验收项（§A16.8），而非锦上添花。

# §A6 屏幕规范

每个屏幕包含：**目的**；**状态读取**（来自引擎/app-services）；**发出的动作**
（每个都是经由 SDK 构建的已签名交易，核心 §15.5/§6）；**后果文本**（核心 §11.4 ——
UI 隐藏复杂度，绝不隐藏后果）；**签名提示**（核心 §11.6 —— 动作、金额、受影响的
底池/状态、确切的字节/意图；不静默签名）；**边缘状态**（loading/empty/error/degraded）；
**无障碍**。屏幕 × 动作 × SDK 矩阵见 Appendix II。

## §A6.1 启动 / 健康
- 目的：展示技术栈启动过程（桌面：§A3.2 生命周期按服务；web：与
  relay+node 的连接），在 `READY` 之前门控游玩。
- 状态读取：`services.status`（桌面）/ 连接状态（web）。
- 动作：无链上动作；仅控制（重试、查看诊断）。
- 后果文本："正在启动本地 node……"、"正在重连（第 n 次尝试）……"、"某服务已降级 ——
  在其恢复前游玩暂停。"
- 边缘状态：降级/致命态显示引导式恢复入口与 `diag.bundleDiagnostics`。
- A11y：状态经由 ARIA live 区域播报。

## §A6.2 钱包（创建 / 导入）
- 目的：建立身份与托管后端（核心 §9）。
- 状态读取：托管后端 + 密钥存在性（UI 中无原始密钥）。
- 动作：无链上动作；密钥操作受托管信任（§A8）；导入/创建流程。
- 后果文本：种子处理警告；"你的密钥从不离开本设备的安全边界"
  （以 §A10 所述信任面为界 —— 软件托管信任设备；如实说明）。
- 边缘状态：导入校验错误 fail closed。
- A11y：机密录入仅键盘、从不记录、从不播报。

## §A6.3 大厅
- 目的：查找玩家；列出/创建/加入牌桌；presence（核心 §8、§15.1）。
- 状态读取：来自 relay 的 presence + 牌桌列表（仅传输/索引，核心 P3）—— 视为
  提示，而非真相。
- 动作：`createTable`、`joinTable`（核心 §15.1）；加入/注资是一笔已签名交易（核心 §6.1
  Funding / Table-mgmt）。签名提示说明被锁定的本金与中止时的退款路径。
- 后果文本："加入会锁定你的本金；若牌桌在开打前中止，则经由
  预签名路径退款。"
- 边缘状态：空大厅；relay 不可达（提供 LAN 模式 / 重试）；陈旧 presence 在
  刷新时被调和。
- A11y：牌桌列表是可导航列表，带有带标签的加入控件。

## §A6.4 建桌 / 配置
- 目的：选择变体、下注结构（NL/PL/FL）、座位、盲注/底注排程、超时配置；
  显示哈希后的配置。
- 状态读取：ruleset 校验（核心 §15.3）；**计算出的 `rulesetHash`**（核心 §5.2、
  REQ-POKER-002）—— 由引擎/SDK 产出，显示给所有玩家，绝不由 UI 计算。
- 动作：`TABLE_CREATE` 绑定到 `gid`+`rulesetHash`（核心 §6.3）。
- 后果文本："此配置被哈希并绑定进每一笔交易；它在手牌中途
  不可更改。"
- 边缘状态：无效 ruleset 以一条具体消息阻止创建。
- A11y：表单控件带标签；显示的哈希为可选中文本。

## §A6.5 牌桌视图（核心玩法屏幕）
- 目的：打这手牌。
- 状态读取：座位、轮到谁、公共/明牌、底池及边池、余额、计时器、以及
  执行状态的**超时默认**文本（核心 §11.4、§19.E）。多方边池来自
  引擎（核心 §5.5、§19.B）；UI 绝不计算它们。
- 动作（每个都是已签名交易；核心 §5.4、§6.1 Action）：`check`/`bet`/`call`/`raise`；
  **弃牌而不揭示**（核心 P5、§4.6、§6.6 Fold）；在拥有该机制的变体中用 `draw`/`stand`（核心
  §7.3）。下注大小经由接口遵循当前下注结构（核心 §5.4）。
- 后果文本（必须精确，核心 §11.4）：例如 "若你什么都不做，将在 30s 后看牌"；"若你
  在面对一注时什么都不做，将在 30s 后弃牌 —— 你永远不会被强迫下注"；"揭示
  翻牌需要每位玩家；若有人扣留，则手牌进入恢复。"
- Board 揭示：呈现为**带超时默认的 N-of-N 协作转换**（核心 §4.6 M2）；
  UI 显示协作释放进度与恢复默认 —— 它不将
  board 揭示呈现为单方面的"发布"。
- 边缘状态：未轮到你（控件清晰禁用）；对手正在超时（倒计时至
  默认）；连接性降级（横幅；canonical 路径仍权威）。
- 签名：每个发出的动作都唤起签名模态框（§A6.7）。
- A11y：回合变化、计时器与后果均被播报；动作控件键盘可操作，
  带清晰的启用/禁用语义。

## §A6.6 手牌查看（私密）
- 目的：玩家查看自己被隐藏的牌。
- 状态读取：**经由托管边界**的本地解密（核心 §11.5、§A8）—— 渲染出的
  牌面仅存于受控查看器路径中；密钥不会越过该路径进入 UI 进程。
- 动作：无。
- 边缘状态：解密不可用（托管后端未就绪）→ 明确告知，无伪造牌。
- A11y：牌面带有文本等价物（点数+花色）供屏幕阅读器使用，仅在查看器路径中。

## §A6.7 签名提示（模态框，跨屏幕）（REQ-APP-060；核心 §11.6）
- 目的：为每笔交易获得明确、知情的同意。
- 显示：动作；金额；受影响的底池/状态；以及正在签名的确切意图/字节。
- 任何地方都不静默签名；无任何会自动签署转移价值动作的"记住我的选择"。
- 边缘状态：用户拒绝 → 不发出动作，状态不变；签名失败（托管错误）→
  明确失败，fail-closed。
- A11y：模态框焦点被困其内、完全键盘可操作、内容被播报。

## §A6.8 摊牌 / 结算
- 目的：最小揭示摊牌与确定性结算。
- 状态读取：谁必须揭示什么（核心 §5.6）；结算结果与最终余额（核心 §5.7、
  §19.B）—— 来自引擎。
- 动作：揭示交易（核心 §4.6、§6.6），随后是收尾/结算花费（核心 §6.6）。
- 后果文本："只有争夺者揭示，且只揭示裁定底池所需的部分。"
- 边缘状态：某争夺者扣留揭示 → 显示恢复超时路径（核心 §6.4）。
- A11y：揭示的手牌与彩金均被播报。

## §A6.9 重连 / 恢复
- 目的：重新加入一张进行中的牌桌；显示恢复状态。
- 状态读取：来自对等方/relay 的 transcript 缺口；引擎重建（核心 §8.6、REQ-NET-007）；任何
  进行中的恢复/超时（核心 §6.4）。
- 动作：请求缺口；恢复。
- 后果文本："正从交易记录重建牌桌 —— 你的视图正在被还原；
  牌桌的真相从不依赖于本设备。"
- A11y：进度被播报。

## §A6.10 Transcript / 重放查看器
- 目的：导出与离线确定性重放（核心 §12.2/§12.3）。
- 状态读取：transcript（有序 valid tx 集 + commit/reveal 材料）。
- 动作：导出；逐步/重放。
- 注：重放是**引擎**逐字节相同地重建该手牌（核心 §12.3、
  REQ-DATA-003）；查看器渲染该重建。核验字节相同是构建运行检查
  （§A16.4），而非 UI 计算。
- A11y：重放控件键盘可操作；每一步的状态被播报。

# §A7 发现、大厅、匹配、连接

依照核心 §8 与决策 D4。应用消费这些网络契约；它不重新定义
协议。

## §A7.1 角色与信任（REQ-APP-070；核心 §8.1、P3）
relay **仅为传输 + 索引，绝非真相之源**。客户端独立
推导状态，并通过从 valid 交易集重建来检测撒谎/有故障的 relay。
indexer 提供投影以方便使用；客户端将其视为待确认的提示。

## §A7.2 发现（Phase 1）（REQ-APP-071；核心 §8.2、D4）
两种机制：一个托管 **relay**（presence、牌桌发现、直消息扇出）与 **LAN
自动发现**（零服务器的同网游玩）。大厅（§A6.3）消费两者。客户端 MUST
在无 internet relay 的 LAN 模式下运作。

## §A7.3 牌桌范围频道（REQ-APP-072；核心 §8.2 Tier B）
加入时，客户端订阅该牌桌的 inventory/object 频道（inv/getdata/object，
Bitmessage 风格）以快速收敛；它除广播动作（canonical 路径）外，
还在该频道发布其动作（speed 路径）。

## §A7.4 双路径发送（REQ-APP-073；核心 §8.3、REQ-NET-003）
每个动作**同时**作为真实交易发往网络，并经由 relay/频道发往牌桌
对等方。speed 路径绝不覆盖 canonical 路径；冲突按
确定性规则解决（核心 §8.5）。连接管理器实现两者并加以调和。

## §A7.5 冲突呈现（REQ-APP-074；核心 §8.5）
每个可执行阶段恰有一个可花费的阶段权利；相冲突的尝试不可能
同时有效。客户端跟随被接受的后继（被下一个被接受的状态转换所引用的那个），
并在阶段关闭前观察到冲突时呈现一个瞬态"resolving"状态；双花尝试
被超时默认弄得在策略上毫无用处（核心 §8.5、
§6.4）。UI 绝不猜测赢家；它显示引擎所接受的内容。

## §A7.6 重连 / 恢复（REQ-APP-075；核心 §8.6）
连接/重连时，客户端从对等方/relay 请求 transcript 缺口，并从 valid tx 集
确定性地重建当前状态；恢复对真相毫无改变，只改变本
客户端的视图。退避与重试上限是构建设置的 `TRACKED ASSUMPTION` 值。

## §A7.7 Phase-5 抽象（REQ-APP-076；核心 §8.7）
internet P2P + NAT 穿透（version/verack/getaddr/addr 对等层）是后续轨道；
连接管理器 MUST 被构建为可让 relay 发现被对等层替换而**无需改动
UI**。现在声明，使 Phase 1 不把 relay 固化为永久真相。

# §A8 通往协议核心的接缝（SDK 消费 + 托管边界）

客户端**不**实现任何密码学、交易构建或游戏逻辑；它消费
SDK（核心 §15.1–§15.9），由核心与适配器 `CT/BS/VA/OB`（核心 §2、§15.8）支撑。

## §A8.1 动作 → SDK 映射（REQ-APP-080；完整矩阵见 Appendix II）
示意性；完整映射归构建，按核心 §15：
- 下注：`getLegalActions`（§15.2）→ `buildAction`（§15.5）→ `custody.sign`（§15.6）→ 双路径发送
  （§A7.4）。
- 弃牌：`buildFold`（§15.5）→ 签名 → 发送。
- Board 揭示：一个 N-of-N 协作转换（核心 §4.6）；UI 显示协作释放 +

- 结算：揭示 tx（§4.6、§6.6）随后收尾花费（§6.6）；引擎计算彩金。

## §A8.2 托管边界（REQ-APP-081；核心 §9.3、§11.5）
牌解密与签名发生在 `Custody` 接口（`derive`、`sign`、
`decryptToViewer`、`combineSignShare`）背后。桌面：Rust 侧或隔离
worker（§A3.1）上的受信托管。Web：一个隔离 worker。UI 仅经由查看器路径接收渲染出的牌面；
原始密钥从不越过到 DOM。后端可插拔（软件默认 AD7；门限 `OB.custody`
FROST/GG20；TEE 可选，稍后）。

## §A8.3 签名模式呈现（REQ-APP-082；核心 §4.3、D9）
当前签名模式（A：揭示时重构，Phase-1 默认；B：门限/不重构）
记录于 ruleset 中；UI 在任何显示之处都呈现**实际的**密钥处理保证
（核心 REQ-CRYPTO-008、P8）。当 Mode A 生效时，UI MUST NOT 声称 Mode B 的"无完整密钥"
属性。

## §A8.4 审计输出边界（REQ-APP-083；核心 §2.3 INV-VA-2）
凡客户端显示 `VA` 支撑的审计/选择性披露输出之处，它都呈现所述
边界 —— 仅包含性/完整性/选择性披露/算术，**而非**源头真相 ——
且不得夸大它（P8）。

## §A9.1 性能 / 延迟（REQ-APP-090）

## §A9.5 可移植性（REQ-APP-094）

# §A10 安全架构（与 Microsoft SDL 对齐）

安全是被设计进去的，而非事后加装。应用层的信任面是
核心所列举之信任面（核心 §18.6）的一次精化；它不在其之上新增任何用户必须信任之物，并
如实陈述其作为。

## §A10.1 应用信任面（REQ-APP-100；精化核心 §18.6）
一个在未受攻陷客户端上的正确用户，在应用层恰好信任：
1. 设备/OS 与托管后端所用的本地安全存储（软件托管 = 设备
   信任；如实陈述，不隐藏）；
2. 已签名安装器 / 所提供之包的完整性（供应链，§A10.6）；
3. 托管边界的隔离性（密钥从不进入 DOM；§A8.2）；
一切建于核心的密码学与共识信任面之上（核心 §18.6）。"无需信任（Trustless）"
处处都不被无条件使用（P8）；行文说出所指的具体属性。

## §A10.2 密钥与机密处理（REQ-APP-101）
密钥仅存于托管边界之内（Rust 侧 / 隔离 worker）。静态时它们被封装
（AEAD；磁盘上绝无明文；绝不用 `localStorage`）。它们从不出现在日志、度量、IPC
载荷或 DOM 中（§A12.5 脱敏）。每局/每张牌标量在 Mode A 中为单局制（核心
§9.2、§4.3），且 UI 呈现这一点。

## §A10.3 查看器/解密边界（REQ-APP-102；核心 §11.5）
解密经由 `decryptToViewer`；渲染出的牌面仅存于受控查看器
路径中。屏幕截取加固/水印是一项**可选的后续轨道**，仅作为部分
缓解而声称（核心 §11.5、P8）—— 绝不声称为对受攻陷客户端的防护。

## §A10.4 输入校验与 fail-closed（REQ-APP-103）
每个跨越信任边界的输入 —— IPC 消息、relay/对等方消息、持久化记录、
SDK 结果 —— 都依其模式校验；无效输入被拒绝并记录，绝不部分
应用。任何含混下的默认是**不**作为（fail-closed）。负向路径被测试
（§A16.6）。

## §A10.5 威胁知情设计（REQ-APP-104；交叉引用核心 §18）
应用层继承核心威胁模型（核心 §18），并新增应用特有条目（每条
带一个界定机制与一个诚实的残余）：恶意 relay（由 P3 + canonical 路径界定，
残余 = 活性降级）、从受攻陷设备提取密钥（由托管
边界/门限/TEE 界定，残余 = 软件模式下的设备信任 —— 已陈述）、受攻陷
客户端上的手牌视图泄露（残余 = 协议无法防止，仅部分缓解）、被篡改的
安装器/包（由签名 + 可复现构建界定，§A10.6）、以及任何
边界上的恶意输入（由 §A10.4 界定）。这些与核心的 `THR-*` 一并编目于 Appendix III/§A18。

## §A10.6 供应链与完整性（REQ-APP-105）
安装器被签名；构建借助锁定的工具链、锁定的 lockfile、以及
记录的制品哈希实现可复现（核心 §10.4、§16；§A14）。依赖被锁定；依赖变更是一次
显式、经审查的事件。构建运行供应链检查（§A16.9）—— Claude Code 的任务。

## §A10.7 网络暴露（REQ-APP-106）
桌面默认将本地服务绑定到 loopback；任何非 loopback 暴露都是显式且
受标志的。relay 客户端按核心协议认证牌桌成员资格；presence 不
泄露手牌信息。

# §A11 错误处理、失败模式、降级运行、恢复

## §A11.1 错误分类（REQ-APP-110；错误码见 Appendix III）
错误是有类型的，且携带一个稳定的码、一条面向用户的消息、一项补救措施、以及一个严重度
（`FATAL`/`DEGRADED`/`RECOVERABLE`/`USER`）。类别：配置、服务生命周期（桌面）、
连接性、托管/签名、协议/校验（一笔被拒绝/无效的交易或状态）、
持久化、以及 UI/渲染。任何错误都不被吞没；每条错误路径都有一个确定的 UI 状态。

## §A11.2 Fail-closed 原则（REQ-APP-111）
一旦含混影响价值或状态，客户端便**不**作为：它不发出
交易，不推进状态，并呈现该状况。静默的尽力而为式猜测是一个
缺陷。

## §A11.3 降级运行（REQ-APP-112）
应用区分"可游玩（playable）"、"降级（degraded）"（例如 relay 宕机但 canonical 路径存活 —— 游玩
带横幅继续；重连运行）、以及"暂停（paused）"（例如桌面上 node 宕机 —— 游玩门控直至
恢复）。每一种都是显式且被播报的（§A12、§A5.5）。

## §A11.4 恢复路径（REQ-APP-113；核心 §6.4）
每个停滞都有路径：决策超时（默认看牌或弃牌，绝不强迫下注）、恢复
超时（扣留的揭示 / 停滞的结算）、以及按 transcript 重连（§A7.6）。客户端
呈现后果与倒计时；推进一个超时默认是任何
对等方都可提交的交易（核心 §6.4）—— 客户端在身为其角色时这样做。

## §A11.5 无价值搁浅不变量（REQ-APP-114；核心 P4）
UI 绝不呈现一种价值会搁浅在合法出口（协作
推进、动作、弃牌、揭示、超时、退款、领奖、收尾）之外的状态。若引擎报告这样一种
状态，则它是一个以 `FATAL` 呈现的缺陷，而非被隐藏。

# §A12 可观测性

## §A12.1 日志（REQ-APP-120）
结构化、分级日志（debug/info/warn/error），按组件，带关联 ID（牌桌/轮次/
动作）。日志默认本地；导出是显式且脱敏的（§A12.5）。

## §A12.2 度量（REQ-APP-121）
针对以下项的计数器/量规/直方图：服务健康与重启（桌面）、连接状态与
重连、动作往返延迟、推导时间、签名延迟、按码的错误计数。这些
喂入 NFR 测量（§A9、§A16.7）—— Claude Code 运行测量。

## §A12.3 追踪（REQ-APP-122）
每个玩家动作贯穿 app-services → SDK → 发送（两条路径）→ 接受的一条追踪，用于诊断
收敛与超时行为。

## §A12.4 审计 / transcript（REQ-APP-123；核心 §12）
transcript（有序 valid tx 集 + commit/reveal 材料）是权威的审计记录；
上述可观测性是运行性的，绝非它的替代。

## §A12.5 脱敏（REQ-APP-124）
日志、度量、追踪、以及诊断包 MUST NOT 含密钥材料、解密的牌面、或
种子/reveal 机密。一个脱敏层强制执行此点；其正确性被测试（§A16.6）。诊断
导出是按需选择且脱敏的。

# §A16 验证与确认 —— 测试架构与完整测试规范

这是**测试设计**：必须测试什么、如何测试、以及验收。**Claude Code 构建并
运行每一项测试；作者不运行任何测试。** 涉及交易/脚本的负向测试 MUST
带 Genesis 规则在**真实的 BSV Script 解释器内部**失败 —— 绝不在包装守卫中，绝不
做签名抽检（核心 §14.3、P9）。

## §A16.1 测试层级（REQ-APP-160）
单元 → 基于属性 → 集成 → **解释器级脚本** → 端到端（E2E）→ 对抗/
故障注入 → 性能 → 安全 → 无障碍。每个层级都映射到 CI 阶段（§A14.2）与
验收闸门（§A18.3）。覆盖率目标是构建测量并报告的 `TRACKED ASSUMPTION`；覆盖率
必要而不充分（一个不演练该行为的通过套件
会被拒绝，P9 精神）。

## §A16.2 单元测试（REQ-APP-161）
纯单元：作为 `(state…) → props` 投影的 view-model（§A5.2）；reducer；持久化
网关的 encode/decode；脱敏层；按执行状态的后果文本选择。每个
组件（§A5.3）都有 loading/empty/error/ready 渲染测试。规范编目于 Appendix IV。

## §A16.3 基于属性的测试与集成测试（REQ-APP-162）
- 属性：确定性 —— 同一有序 valid-tx 集 + ruleset 在多次运行间及两个外壳间
  产出逐字节相同的派生状态（核心 §14.2、REQ-TEST-002）；按缺口重连产出与
  连续游玩相同的状态；UI 绝不为给定的引擎状态启用一个非法动作。
- 集成：`ui-core` + `app-services` + SDK，配以针对 CT/BS/VA/OB 的**契约绑定**伪件 ——
  即一套契约一致性套件，伪件与**真实**适配器皆通过（核心 §2.6
  REQ-DEP-003/004），故对伪件的绿色运行不能认证一个错误的引擎。安全关键
  路径（洗牌、reveal 单次使用、公平博弈、签名）针对**真实**
  实现测试，绝不用伪件（核心 REQ-DEP-004）。

## §A16.4 确定性与重放测试（REQ-APP-163；核心 §12.3、§14.2）
一份 transcript 离线重放至逐字节相同的状态；任何分歧都是缺陷。跨客户端
一致性：两个引擎（桌面与 web 构建）给定同一 valid tx 集 + ruleset 完全一致。
Claude Code 运行这些；作者规定等价关系与语料库。

## §A16.5 解释器级脚本测试（REQ-APP-164；核心 §14.3、P9）
对应用导致被花费的每个脚本模板（funding、action、fold、reveal-or-timeout、
settlement、fair-play、可选 TTP —— 核心 §6.6），正向花费被接受，而一组
负向花费带 Genesis 规则在真实解释器**内部**失败。应用层的义务
是确保其流程只在顺利路径上构造解释器接受的花费，并且
每个 UI 可达的畸形尝试都被解释器捕获，而非被 UI 守卫捕获。**Claude
Code 运行解释器；作者规定这些用例。**

## §A16.6 健壮性 / 负向 / 故障注入测试（REQ-APP-165；核心 §14.6）
在每个阶段断连；陈旧/重复动作；超时竞态；扣留的揭示（玩家与
board）；相冲突的花费；mempool 驱逐 + 重提交；损坏的持久化记录；畸形的 IPC/
relay/对等方消息；托管/签名失败；手牌中途 WebView/标签页崩溃；桌面上
node/relay/indexer 崩溃。每条都映射到一个 `REQ-*` 与一个确定性的预期 UI+状态结果（fail-closed、恢复、
或默认）。脱敏被故障测试（任何机密都绝不出现在日志/度量/追踪/导出中）。

## §A16.7 性能测试（REQ-APP-166；§A9）
构建在一台参考机器上**测量** §A9 的目标（动作往返、帧预算、推导时间 vs N、
内存上限），并经由 `reproduce` 将它们记录为可复现的数字
（核心 §14.5、P10）。作者不产出这些数字；构建产出。

## §A16.8 无障碍测试（REQ-APP-167；§A5.5）
自动化 a11y 检查（角色/标签/对比度/焦点）外加对每个屏幕的脚本化纯键盘与屏幕阅读器
走查；回合/计时器/后果播报被核验。无障碍是一道闸门
（§A18.3），非可选项。

## §A16.9 安全测试（REQ-APP-168；§A10）
托管边界测试（无密钥穿越 IPC/DOM/日志）；每个边界上的 fail-closed 输入校验测试；
供应链检查（锁定依赖、可复现构建、制品哈希核验）；
安装器签名核验。由 Claude Code 运行。

## §A16.10 端到端 / 验收（REQ-APP-169；核心 §14.4、§14.7）
一手脚本化的完整单挑 Hold'em 在 regtest 上贯穿自包含镜像运行：setup →
shuffle → deal → 各下注轮 → showdown/settlement，外加 §A16.6 的故障注入，全部
产出可复现的 transcript —— 在桌面构建与 web 构建上皆然。这是
Phase-1 验收 E2E（§A10/§A18.3）。

# §A18 需求登记册、可追溯性、验收闸门

## §A18.1 登记册（REQ-APP-180）
本文档中的每个 `REQ-APP-*` 都是一份机器可读登记册
（`/spec/app-requirements.yaml`）中的一条，带：id、文本、归属模块（§A2–§A8）、验证方法（§A16
层级）、以及状态。该登记册在 CI 中生成/校验，并与唯一的
`REQ-APP-*` 计数保持步调一致（不匹配即为缺陷）。它从不重复定义某条核心需求；某条
精化了核心需求的应用需求会引用它（例如 REQ-APP-030 精化核心 REQ-VM-007）。

## §A18.2 可追溯性矩阵（REQ-APP-181）
每条需求 → 归属模块 → 测试 → 验收闸门。若任何需求没有
验证测试，或共识/安全路径中的任何应用源文件未被追溯，则 CI 失败
（对应核心 §13.3）。

## §A18.3 验收闸门（REQ-APP-182）
一个阶段仅在以下情形被接受：该阶段的每条需求都可追溯到通过的测试；`reproduce`
在镜像内绿色；该阶段的对抗、无障碍与安全测试通过；阶段
E2E（§A16.10）在桌面与 web 构建上皆干净运行；且该
阶段的文档集（§A15）是现行的。任何阶段都不在一个看似绿色却不演练该行为的测试上推进
（P9 精神）。

## §A18.4 完成的实用定义（REQ-APP-183）
应用规范在以下情形完成：Claude Code 能构建 Phase 0 与 Phase 1 **而无需再问一个
进一步的应用设计问题**，每个 `REQ-APP-*` 都已登记并追溯，每项 NFR/
性能/安全/无障碍义务都被构建测量（或携带一个有据可查、待构建运行的
`TRACKED ASSUMPTION`），且文档集已产出且现行。

# §A21 多游戏平台架构

该平台是一个**多游戏**无荷官纸牌游戏应用 —— 所有扑克变体外加 Blackjack，
置于一个游戏模块接口之后。本节规定该设计的**应用层**；它
消费核心游戏模型（核心 §5、§7）与 `GameModule` 接口（核心 §7.1），并新增
变体感知的 UI、大厅/注册表集成、以及逐游戏的测试与推出义务。UI 中不实现任何游戏
逻辑（§A2.2）；UI 在能做到处与变体无关，且仅在某变体的表面确有差异处才作
变体专门化。

## §A21.1 应用层的游戏模块抽象（REQ-APP-210）
每个游戏是一个实现核心 `GameModule` 契约（核心 §7.1：
`init`/`getLegalActions`/`apply`/`isTimeoutEligible`/`isHandComplete`/`settle` + 序列化）的模块，
按核心 §16 打包（`/packages/game-holdem`、`/packages/game-omaha`、`/packages/game-stud`、
`/packages/game-draw`、`/packages/game-razz`、以及计划中的 `/packages/game-blackjack`）。应用
**仅**经由 SDK/引擎消费模块（核心 §15.2）；`ui-core` 从不直接 import 一个游戏模块
（REQ-APP-211）。UI 渲染 `getLegalActions`/`deriveState` 所返回的一切；合法动作的集合
与牌桌形态归模块，被通用地呈现。

## §A21.2 游戏注册表与大厅集成（REQ-APP-212）
一个**游戏注册表**列出可用游戏，每个游戏带：显示名、支持的座位范围（核心
D2：2–9）、支持的下注结构（核心 §5.4：NL/PL/FL）、强制下注模型（盲注 vs
底注+bring-in）、以及牌桌视图变体配置（§A21.3）。大厅（§A6.3）与建桌
（§A6.4）屏幕由此注册表驱动：玩家挑选一个游戏，随后注册表约束
其余配置（座位、结构、盲注/底注）。注册表是**数据**，而非 UI 分支；
添加一个游戏就是添加一条注册表条目 + 一个模块，而非编辑屏幕（REQ-APP-213）。

## §A21.3 逐变体牌桌视图配置（REQ-APP-214）
牌桌视图（§A6.5）是**一个**由来自注册表的变体配置参数化的屏幕：

| Seven-Card Stud | 2 张暗牌 + 明牌 | 无 | **明牌**公开 N-of-N；暗牌私密 | **由 board 驱动**（最高 board 先行动；bring-in = 最低明牌，核心 §7.3.2） | 逐座位明/暗布局；bring-in 指示器；8 人时牌堆耗尽的共享公共明牌 |
| Five-Card Draw | 5 张隐藏 | 无 | 仅摊牌 | 相对庄家按钮 | **换牌控件**：弃 0–5 张不揭示 + 重抽（数量公开，身份私密，核心 §7.3.3）；停牌 |
| Razz | 2 张暗牌 + 明牌 | 无 | 明牌 N-of-N | 由 board 驱动，**反向**（最低先行动；bring-in = **最高**明牌，核心 §7.3.4） | 低牌指示器；反向顺序提示 |
| Blackjack | 独特模型 | — | 独特模型 | — | 见 §A21.7 |

UI 消费配置 + 引擎的逐状态输出；它不编码规则。配置覆盖
完整的 2–9 座范围（§A21.6）。

## §A21.4 三种牌 UI 原语（REQ-APP-215；核心 REQ-FSM-003/004/005）
- **暗牌** —— 由 `draw(position)` 发出，随后经由托管边界**私密揭示给其持有者**
  （§A8.2）；他人看到一张隐藏的牌，持有者看到一张查看器路径牌面。
- **明牌** —— 由 `draw(position)` 发出，随后**由一个带超时默认的 N-of-N
  协作转换立即公开揭示**（核心 §4.6）；UI 显示协作释放
  进度，而非单方面发布。
- **换牌**（five-card draw）—— 玩家**不揭示**地将所选隐藏牌交付给一手死手
  （一次部分弃牌），并被发以相同数量的全新隐藏牌（私密揭示给
  换牌者）；**数量公开，身份不公开**。UI 暴露一个多选弃牌
  + "停牌"（换 0 张），各自经由 SDK 产出相应的已签名交易。
每个原语都是一笔或多笔已签名交易（核心 §6.1）；UI 获取签名同意（§A6.7）
且绝不揭示规则所保持隐藏之物。

## §A21.5 下注结构 UI（REQ-APP-216；核心 §5.4）
下注计量器经由 `BettingStructure` 接口（核心 §5.4）感知结构：No-Limit
（max = stack）、Pot-Limit（max = pot + call）、Fixed-Limit（固定小/大注，加注有上限）。UI
读取 `legalBets` 且仅渲染合法的下注大小；它绝不计算其范围。Stud/Razz 小/
大注下注轮与 Fixed-Limit 明对规则由引擎呈现。

## §A21.6 多方（3–9 座）UI（REQ-APP-217；核心 D2、§5.5、§19.B）
座位环、行动顺序指示器、与**边池显示**处理 2–9 座。边池（主池 +
有序边池，核心 §5.5/§19.B）由引擎计算并被渲染；UI 显示每个底池、
其有资格的座位、与彩金。行动顺序逐游戏而定（相对庄家按钮 vs 由 board 驱动，
§A21.3）。Phase 1 固定为 2 座（核心 D1/D2）；UI 与状态模型处理完整范围
而无需重新设计（REQ-APP-218）。

## §A21.7 Blackjack —— 必需游戏，独特的无荷官模型（REQ-APP-219；核心 D7）
Blackjack **是一个必需游戏**（R6），并在此被作为一等的**计划**模块捕获 —— 它
未被弃置。但无荷官 blackjack **不是**对称的 mental-poker 洗牌：它是
玩家对庄家，而一个无荷官构造需要其**自己的**隐藏与结算模型
（核心 D7 禁止将该差异偷带进扑克管线）。因此 Blackjack 是一个
带自身发牌/结算设计的 `GameModule`，且其协议模型为 **DECISION REQUIRED** —— 一条
专门的设计轨道（与扑克 §7.3 工作并行），涵盖在无受信荷官下
如何实现"庄家"角色、如何隐藏并发出牌靴、以及结果如何结算。应用层
现在预留：其注册表条目；其变体配置（要牌/停牌/加倍/分牌/保险控件、一个
庄家区域、无玩家间底池）；及其测试义务。该模块在其协议
模型确定**之后**才交付。这在不过度声称项目尚未规定之模型的前提下，
捕获了该需求与该接缝（P7/P8）。

## §A21.8 逐游戏测试义务（REQ-APP-220；Claude Code 运行）
对每个游戏模块：变体状态机在 regtest 上做 E2E 演练（核心 §7.3、§19.E + 那些
多方泛化）；手牌评估路径针对核心的**生成的**
向量（核心 §19.D —— Omaha 2+3、ace-to-five low 等）测试，那些向量由 **Claude Code 通过运行
预言机生成**；变体牌桌视图配置（控件、揭示、顺序）在每个支持的座位
数上、于桌面与 web 构建上皆被测试。作者规定这些用例；Claude Code 运行它们。

## §A21.9 多游戏分阶段推出（REQ-APP-221；对应核心 §17）
- **Phase 1** —— Texas Hold'em（单挑 NL，regtest，含发现）：完整管线（核心 D1）。
- **Phase 2** —— Hold'em 健壮性 + 多方（6-max）+ 公平博弈（核心 §17 Phase 2）。

# §A22 NFT / 吊销集成轨道（必需；后续轨道；接缝已暴露）

真正可转让 + 可吊销的 NFT 是一项**必需能力**（R10）：一旦 Bob 拥有该物品，Alice
便永久失去访问。平台经由 NFT/访问接缝（核心 §15.7）在 `overlay-broadcast`（核心 §2.4）上
实现这一点，作为一条**独立的产品轨道** —— **不**在核心扑克
阶段内交付，但暴露其接缝以使其无需重新架构即可集成（核心 §0.3、§15.7）。

## §A22.1 吊销实际为何（REQ-APP-230；核心 §2.4、P8）
吊销 = 一个由任何运营者都不裁决的**未花费到期输出**（核心 §2.4 INV-OB-2）：访问是一个
有资金的会话，其续期花费一个成员输出；**过期仍未花费 = 已吊销**。内容
在一个密钥图（Logical Key Hierarchy，`O(log n)` 重新密钥）下加密；一次转让重新密钥，使先前
拥有者的密钥不再能打开该内容。UI MUST **精确**陈述该属性 —— 吊销是
链上的到期/重新密钥事实；平台所声称的不超过 `overlay-broadcast` 所强制的（P8）。
在吊销**之前**被外泄的副本在任何此类系统所能吊销的范围之外；UI 在
相关处如此说明。

## §A22.2 应用接缝与 UI（REQ-APP-231；核心 §15.7）
SDK 接缝（`OB` 支撑的 `accessSession`、`revokeByExpiry`、`wrapContentKey`，核心 §15.7）被暴露
但在核心扑克阶段中未使用。NFT 产品 UI（陈列馆；上架/购买/出售/**带吊销的
转让**）是一组后续轨道屏幕；转让触发重新密钥，使旧拥有者的访问在链上
被吊销。市场经济学在此范围之外。

## §A22.3 边界与测试（REQ-APP-232）
Grok 文档中的 `revocable-nft-tee` 仓库**不**在作者已发布的清单中（核心 §2.4 注）；
可吊销内容能力即 `overlay-broadcast` 原语，而打磨完成的 NFT 产品
尚未交付 —— 平台在该原语之上构建它、仅声称它所强制之物、并针对
**真实的** `overlay-broadcast` 实现测试 转让→吊销（核心 §2.6 REQ-DEP-004）。
TEE 封装托管是可选的（AD7），且吊销并不需要它。

# Appendix I — IPC 命令/事件目录（桌面 Tauri main ⇄ ui-core）

记法：`cmd(req) → res` 表示请求/响应；`evt: {…}` 表示单向事件。类型描述
**消息契约**（非代码）；确切定义按 `protocol-types` 与核心 §15 构建。
每个命令遵守一条信任边界规则（§A10.4）与 fail-closed（§A11.2）：一条无法识别/畸形的
消息被拒绝（`IPC-MALFORMED`，Appendix III）并记录，绝不部分应用（REQ-APP-026）。
原始密钥材料 MUST NOT 出现在任何载荷中（REQ-APP-025）。在 web 上，同一契约由
连接层针对远程服务提供（§A4.1）；`services.*` 仅限桌面。

### I.1 `services.*`（仅桌面）—— 服务生命周期（§A3.2）
- `services.start() → {accepted: bool}` —— 有序启动（node→indexer→relay）；幂等。
- `services.stop() → {accepted: bool}` —— 反序关闭。
- `evt services.status: {service:'node'|'indexer'|'relay', state:'starting'|'healthy'|'degraded'|'failed'|'recovering', detail?:string, attempt?:int}` —— 在每次转换时发射。
- 信任边界：仅控制面；无价值，无密钥。错误：`SVC-START-FAILED`、`SVC-PORT-CONFLICT`、`SVC-STORE-CORRUPT`。

### I.2 `chain.*` —— 链访问（透传至 `Chain`，核心 §15.9）
- `chain.broadcast(rawTx:hex) → {txid:hex, status:'accepted'|'seen'|'double-spend-attempted'|'rejected', reason?:string}` —— 广播一笔**已签名**的 tx；从不构建或签名。
- `chain.outpointStatus({txid,vout}) → {state:'unspent'|'spent'|'unknown', spentBy?:hex}`。
- `chain.txStatus(txid) → {state:'mempool'|'confirmed'|'evicted'|'unknown', confirmations?:int}`。
- `chain.headers({from?,count?}) → {headers:BlockHeader[]}` —— SPV 边缘检查（核心 §34）。
- `chain.conflicts(outpoint) → {conflicting:hex[]}` —— 双花可观测性（核心 §8.4）。
- 信任边界：仅读取/广播；响应是待确认的提示（核心 P3）；`accepted` ≠ 终局（终局 = 确认深度，核心 §5.7）。错误：`CHAIN-UNREACHABLE`、`CHAIN-BADTX`、`CHAIN-TIMEOUT`。

### I.3 `custody.*` —— 托管受信操作（仅结果；核心 §9.3、§15.6）
- `custody.derive({gid,j?,role}) → {publicKey:hex}` —— 仅 HKDF 派生的**公**钥；绝不返回标量（REQ-APP-025）。
- `custody.sign(intent:SignIntent) → {signature:hex}`，其中 `SignIntent={sighashPreimage:hex, describe:{action,amounts,potOrState}}` —— 恰好签名提示所渲染的字节（§A6.7）；不匹配时 fail-closed。
- `custody.decryptToViewer(card:ConcealedCardRef) → {viewerToken:opaque}` —— 解密进受控查看器路径（§A10.3）；返回一个查看器令牌，绝非 DOM 明文。
- `custody.combineSignShare(req) → {share|signature}` —— Mode-B 门限份额（核心 §6.7）；Mode A 下不存在。
- 信任边界：通往密钥操作的**唯一**之门；密钥从不越过它。错误：`CUSTODY-LOCKED`、`CUSTODY-BACKEND-UNAVAILABLE`、`CUSTODY-INTENT-MISMATCH`、`CUSTODY-DECRYPT-FAILED`。

### I.4 `config.*`
- `config.runtime() → {ports, network:'regtest'|'mainnet', dataDir, flags}`。
- `config.setNetwork(network) → {accepted:bool}` —— **受守护**：除非主网研究标志已设，否则拒绝非 regtest（§A3.5、REQ-APP-030）；成功需要 UI 横幅。错误：`CONFIG-MAINNET-FLAG-REQUIRED`、`CONFIG-INVALID`。

### I.5 `diag.*` —— 诊断（已脱敏，§A12.5）
- `diag.logs(query) → {entries:LogEntry[]}`（已脱敏；无密钥/牌面/机密，REQ-APP-124）。
- `diag.metrics() → {metrics:Metric[]}`（§A12.2）。
- `diag.bundleDiagnostics() → {path}` —— 按需选择，已脱敏。错误：`DIAG-EXPORT-FAILED`。

# Appendix IV — 测试用例目录（核心集；完整的逐用例展开在 Pass A5）

每个用例包含：id、层级（§A16.1）、闸门/前置条件、预期 UI+状态、分类、所验证的
需求。**Claude Code 运行每个用例**（§A19）。涉及脚本/交易的负向用例带 Genesis 规则在
真实解释器**内部**失败（核心 §14.3、P9）。下方的集合解决了 Appendix III 引用的每个 TC-*。

### IV.1 确定性与重放（§A16.4）
- **TC-DET-001**（属性）同一有序 valid-tx 集 + ruleset ⇒ 多次运行间逐字节相同的派生状态 —— REQ-APP-163、核心 REQ-TEST-002。
- **TC-DET-002**（属性）桌面与 web 构建在同一集合上完全一致 —— 跨客户端一致性。
- **TC-DET-003**（集成）按缺口重连 ≡ 连续游玩 —— REQ-APP-075。

### IV.2 引擎合法性 / UI（§A16.2）
- **TC-UI-001** UI 绝不启用一个不在 `getLegalActions` 中的动作 —— REQ-APP-010/051（fail-closed）。
- **TC-UI-002** 后果文本与执行状态的超时默认相符（看牌 vs 面对一注时弃牌；绝不强迫下注）—— 核心 §6.4。

### IV.3 错误路径（§A16.6）—— 解决 Appendix III TC-ERR-*
- **TC-ERR-001** 无效配置 ⇒ FATAL，无法游玩。**TC-ERR-010/011/012** 启动失败/端口/损坏存储 ⇒ DEGRADED + 引导式恢复。**TC-ERR-020/021/022** 链不可达/badtx/超时 ⇒ 横幅+speed 路径 / fail-closed 重新推导 / 退避。**TC-ERR-030** relay 不可达 ⇒ LAN/对等方回退。**TC-ERR-040/041/042** 托管锁定/不可用/解密失败 ⇒ 解锁 / 暂停签名 / 无伪造牌。**TC-ERR-050** 无效动作 ⇒ 重新推导，仅合法。**TC-ERR-060/061** 损坏记录/配额 ⇒ 隔离 / 导出-裁剪。**TC-ERR-070** 渲染故障 ⇒ 错误边界恢复一个屏幕。

### IV.4 安全（§A16.9）—— 解决 TC-SEC-*
- **TC-SEC-001** 任何日志/度量/追踪/包中无密钥材料（脱敏，REQ-APP-124）。**TC-SEC-002** 无密钥穿越 IPC 或到达 DOM（REQ-APP-025/081）。**TC-SEC-005** 签名意图 ≠ 所呈现字节 ⇒ 拒绝（CUSTODY-INTENT-MISMATCH）；无自动重试。**TC-SEC-010** 畸形 IPC ⇒ 拒绝+记录，绝不部分应用。**TC-SEC-020** 无标志的主网 ⇒ 被拒绝 + 受标志时显示横幅。**TC-SEC-030** 安装器签名 / 制品哈希核验（供应链，§A10.6）。

### IV.5 对抗 / 故障注入（§A16.6）—— 解决 TC-ADV-*
- **TC-ADV-001..006** 在每个阶段断连（发牌/下注/揭示/摊牌/结算/恢复）⇒ 正确的默认/恢复；无搁浅价值。**TC-ADV-007** 相冲突的花费 ⇒ STATE-CONFLICT 解决到被接受的后继（核心 §8.5）。**TC-ADV-008** 扣留的 board/玩家揭示 ⇒ N-of-N 超时默认进入恢复（核心 §4.6/§6.4）。**TC-ADV-009** 陈旧状态微支付广播 ⇒ 1 聪保证金被没收（核心 §18.1）。**TC-ADV-010** mempool 驱逐 ⇒ 重提交；收敛恢复。**TC-ADV-020** 任何会搁浅价值的状态 ⇒ STRANDED-VALUE FATAL，被呈现（核心 P4、§A11.5）。

### IV.6 解释器级脚本（§A16.5；Claude Code 运行真实解释器，Genesis）
- **TC-INT-001** 每个模板的正向花费被接受（funding、action、fold、reveal-or-timeout、settlement、fair-play、可选 TTP —— 核心 §6.6）。**TC-INT-002** 每个模板的负向组在解释器**内部**失败（非包装守卫；无签名抽检）：换牌替换被字节相等拒绝（核心 §6.3）、坏的 reveal 开启被拒绝、被重放的分支被绑定拒绝（核心 §6.3）。

### IV.7 无障碍（§A16.8）

### IV.8 逐游戏（§A21.8）—— 每变体一块

# Appendix VI — 术语表（应用层）

与核心术语表一致（核心 §0.6）。
- **Supervisor（监管器）** —— 派生/监管/拆除本地服务并
  承载托管受信操作的 Tauri Rust main 进程（§A3）。
- **Shell（外壳）** —— UI 核心的轻薄宿主：`client-desktop`（Tauri/WebView2）或 `client-web`（Vite）；无
  业务逻辑（§A5）。
- **UI 核心（`ui-core`）** —— 两个外壳都运行的共享 TS/React 核心：store、view-model、组件（§A5）。
- **app-services** —— 位于 `ui-core` 与 SDK 之间的客户端层：连接管理器、
  同步/推导器、托管客户端、超时/观察者、持久化网关（§A2.3）。
- **Connection manager（连接管理器）** —— 拥有两条发送路径与牌桌频道/presence/重连生命周期（§A7）。
- **Canonical path（canonical 路径）** —— 将动作作为真实交易发往 node/网络（权威；
  最终终局）（§A7.4，核心 §8.3）。
- **Speed path（speed 路径）** —— 将动作直接发往牌桌对等方以快速收敛；绝不覆盖
  canonical 路径（§A7.4，核心 §8.3）。
- **Table channel（牌桌频道）** —— 逐牌桌的 inventory/object relay 订阅（§A7.3，核心 §8.2 Tier B）。
- **Custody boundary（托管边界）** —— 密钥所居且签名/解密所发生的受信进程/worker；密钥
  绝不越过它到 DOM（§A8.2，核心 §9.3/§11.5）。
- **Viewer path（查看器路径）** —— 仅向其持有者渲染一张解密牌面的受控路径；绝非 DOM
  明文（§A10.3，核心 §11.5）。
- **Fail-closed** —— 一旦含混影响价值/状态，便不作为并将其呈现（§A11.2）。
- **Degraded / Paused（降级 / 暂停）** —— 降级 = 游玩在连接性降低下继续（canonical 路径存活）；
  暂停 = 游玩门控直至某个所需的本地服务恢复（§A11.3）。
- **Self-contained image（"VM"，自包含镜像）** —— 捆绑
  node+relay+client 并带一键 bootstrap 的可复现容器（及可选 VM 镜像）（§A14，核心 §10/D5）。
- **Game module / variant profile（游戏模块 / 变体配置）** —— 一个 `GameModule`（核心 §7.1）与
  为该游戏参数化牌桌视图的注册表数据（§A21）。
- **TRACKED ASSUMPTION** —— 构建必须达到并**测量**的值/目标，而非已达到的声称（§A0.1）。
- **DECISION REQUIRED** —— 一个尚未确定的选择；在确定之前处处都不断言（§A0.1）。
