## 1. 硬性规则（不可商量 —— 来自委托人；违反任何一条即为失败）

1. **零捏造。** 每个数字都追溯到一个被引用的来源、一个被测量的数值，或被
   标记为 `TRACKED ASSUMPTION`。一个造成内部不一致的硬编码数字是一个
   实质性缺陷。**通过运行代码生成向量，绝不凭记忆**（见 §3，
   预言机）。一个编造的引用或数字被视为欺诈。
2. **无假设；对其分类。** 每个假设都显式出现。隐藏
   假设是最糟糕的失败类别。如果某件事未确定，将其标记为
   `DECISION REQUIRED` 或 `TRACKED ASSUMPTION` —— 不要断言它。
3. **不夸大。** 绝不声称超过构造所保证的内容。白纸黑字陈述边界
   （例如，审计系统无法在源头检测谎言；N-of-N 揭示是一个活性
   风险；"无信任"仅在所枚举的信任面 §18 之内）。
4. **仅限 BSV，后 Genesis。** 没有 BTC 代码，没有 BTC 假设，没有 Lightning 式的脚本内
   惩罚。**`OP_CHECKLOCKTIMEVERIFY`/`OP_CHECKSEQUENCEVERIFY`（CLTV/CSV）在后 Genesis 的
   BSV 上是空操作** —— 计时在**交易层级**强制执行（在原始替换规则下使用
   `nLockTime` + `nSequence`）。混淆 BTC 和 BSV 即为失败。
5. **工程标准。** NASA **NPR 7150.2** 保障实践 + 一份**有文档记录的 Power-of-Ten
   适配**（不是字面合规 —— 规则 3 和指针规则在 GC
   运行时中为 N/A；明确说明）。需求可追溯性矩阵。**解释器层级测试**：脚本
   花费通过真实的 BSV Script 解释器以 Genesis 规则运行；**否定测试
   必须在解释器内部失败**，绝不在包装器守卫中失败；签名抽查
   不可接受。**可复现向量**：一条 `reproduce` 命令重新生成每一个数字
   并在不匹配时以非零状态退出。
6. **风格。** 直接、简洁、无歧义。**禁止道歉。** 不要灌水。不要
   在没有证据的情况下虚张声势或听起来自信。
7. **不要质疑委托人陈述的事实。** 委托人已经陈述
   `prof-faustus` 仓库存在，且 Claude Code 可以访问它们。**接受
   这一点。** 针对它们的契约进行构建（规格说明的 §2.1–§2.4）。不要重新核验它们的
   存在，也不要重新争论它。

## 2. 已核验的事实（引用这些；不要重新推导或重新检查）

**依赖仓库（真实的；本平台通过规格说明 §2 中的适配器契约
消费它们）：**

- `bonded-subsat-channel` —— 亚聪通道（粒度 `k`，最大余数 `Q*`
  整聪结算，固定 1 聪反作弊保证金）**以及**一个自包含的嵌入式
  BSV 系统（原生 P2P 线路节点、按工作量最长的 PoW 区块头链、DB 支撑的区块/UTXO
  存储、在原始替换规则下检测冲突的内存池、HD 钱包）。这个
  嵌入式节点是本平台用于 VM 的**本地链后端**（规格说明 §10.2，D6）。
- `verifiable-accounting`（Rust）、`verifiable-accounting-bsv`（TS）、`verifiable-accounting-chain`
  （TS）—— 针对区块头 `merkleroot` 的 Merkle 包含证明；选择性披露；
  triple-entry。**已陈述的边界（必须浮现出来，绝不夸大）：** 仅建立
  包含性、完整性、选择性披露、对已披露记录的算术正确性；
  **不**建立源头真实性。当承诺必须保留在花费链接图中时，将其作为**活动
  脚本中的 pushdata，而非 `OP_RETURN`** 来承载。
- `overlay-broadcast` —— 密钥图（逻辑密钥层次结构）广播加密、`O(log n)`
  重新密钥、吊销 = **未花费的过期输出**、FROST/GG20 门限托管、ECIES /
  AES-256-GCM。**已按 NPR 7150.2 + Power of Ten 分级，带一个需求可追溯性
  矩阵**（本平台采用相同标准）。是后续可吊销内容 NFT（一个
  独立产品，不在核心扑克阶段内）以及底池密钥可选门限托管的基础。
- `cardtable` —— mental-poker 引擎 + 交易原生的游戏基底。**状态（作者
  自己的话）：** 协议已完整规格化；离线密码学 + 仿真可运行；
  早期构建阶段大体到位；**完整的多牌游戏部分完成；首个目标
  游戏是 In-Between**；CLI 优先的研究代码；默认 regtest。本平台使用其
  *原语*，而非其游戏，并且**不**假设其中存在 Hold'em。

**对 Grok 撰写文档的更正（不要传播其错误）：** `cardtable`
**不**包含 5 种扑克变体 + Blackjack（"阶段 0–5 完成"是假的）。仓库
`revocable-nft-tee` 和 `triple-entry-bsv-sql` **不**在作者当前发布的
列表中；可吊销 NFT 构建于 `overlay-broadcast` 之上，且打磨后的产品尚未交付。

**专利 GB2616862（洗牌/结算原语；已完整阅读；在记忆中）：** "Set
shuffling"，Burns & Wright，申请人 nChain Licensing AG，于 2022-03-22 提交，公布为
**GB2616862A**（申请）2023-09-23。引用为 **A**，除非/直到 B 授权被确认
（v27 论文引用了一个"B"，未被文档确认）。机制：洗牌密钥
`P'=(s,±√(s³+7))` 在 secp256k1 上，私钥 `s` = x 坐标，公钥 `P=s·G`；逐元素
**组合密钥** `Q_j = Σ P_{p,j}`（点加法）；两轮加密（共享标量，
然后逐元素的不同标量）+ 重排 = 洗牌，在揭示时通过逆标量
反转；OP_RETURN 阶段承诺；结算通过 2-of-2 出资多签 + nLocktime
退款 + 按 locktime 分级的下注交易 + 公平博弈脚本 + 可选的 2-of-3 TTP。已给出的示例
是 **2 方 / 3 牌**（A>K>Q）；扩展到 N 方/52 牌是本平台的工作。

**签名模式事实（BLOCKER B1 已解决 —— 不要重新引入该错误）：** GB2616862
在揭示时**重建**组合私钥 `w_j`（每方披露其标量；
获胜者求和并签名）。早先规格说明中"从不在某一处被重建"的论断是
*v27 门限 ECDSA* 路径的属性，**而非**该专利的属性 —— 它是一处内部
不一致，已被移除。规格说明现在声明 **模式 A**（专利字面，
揭示时重建；阶段 1 默认；单局密钥；有界手牌窗口暴露）对比
**模式 B**（门限/无重建；升级版；非 BSVM 特有，故 BSV 兼容）。
**绝不在呈现模式 A 的同时声称模式 B 的属性。**

**后 Genesis Script（上文规则 4）：** CLTV/CSV 为空操作；计时在交易层级。
